# Спецификация: создание колонки из меню колонки и защита последней колонки

## 1. Назначение документа
Документ фиксирует обязательные требования к управлению колонками на экране доски.

Цель доработки:
- убрать отдельную кнопку `+ Колонка` из строки управления доской;
- перенести создание колонки в меню действий конкретной колонки;
- добавлять новую колонку строго после той колонки, из меню которой запущено создание;
- запретить удаление последней оставшейся колонки на доске;
- сохранить поведение без двусмысленности для UI, прав доступа и серверной валидации.

В документе нет кода. Все решения ниже являются обязательными.

## 2. Приоритет документа
Для сценариев создания и удаления колонок этот документ имеет приоритет над более ранними формулировками, где создание колонки было привязано к отдельной кнопке в строке управления доской.

После принятия этой спецификации любые упоминания кнопки `+ Колонка` в control-row считаются устаревшими.

## 3. Объект доработки
Под доработкой в этом документе понимаются:
- меню действий колонки;
- модальное окно создания колонки;
- серверная логика создания колонки;
- серверная логика удаления колонки;
- правила видимости и доступности действий по правам.

Вне объекта доработки:
- drag-and-drop колонок;
- drag-and-drop карточек;
- логика редактирования карточек;
- логика типов колонок;
- визуальная тема доски;
- права доступа, не связанные с `columns.create`, `columns.rename`, `columns.delete`.

## 4. Термины
В этом документе используются следующие термины:

- Исходная колонка: колонка, из меню которой пользователь запустил действие создания новой колонки.
- Новая колонка: колонка, создаваемая через действие добавления из меню исходной колонки.
- Логический порядок колонок: порядок колонок доски по `position` от меньшего значения к большему.
- Последняя колонка доски: ситуация, в которой на доске существует ровно одна колонка.

## 5. Целевое поведение строки управления доской
На экране доски отдельная кнопка `+ Колонка` больше не отображается.

Для пользователя не должно существовать второго, параллельного entry-point создания колонки в control-row.

Строка управления доской не меняет свою высоту из-за удаления этой кнопки:
- если в строке есть другие элементы, они продолжают жить в ней;
- если других элементов нет, строка всё равно сохраняет высоту, заданную общей layout-спецификацией.

## 6. Целевое поведение меню колонки

### 6.1. Когда показывается кнопка меню
Кнопка меню колонки `⋮` отображается, если для текущего пользователя доступно хотя бы одно из действий:
- создание колонки;
- изменение названия и типа колонки;
- удаление колонки.

Если у пользователя нет ни одного из этих прав, кнопка меню колонки не отображается.

Право на перестановку колонки не влияет на показ кнопки меню, потому что перестановка остаётся отдельным сценарием.

### 6.2. Состав меню
Пункты меню отображаются в таком порядке:

1. `Добавить колонку после…`
2. `Изменить название и тип…`
3. `Удалить колонку…`

Пункт отсутствует, если у пользователя нет соответствующего права, кроме специального случая для удаления последней колонки, описанного ниже.

### 6.3. Правила видимости пунктов
- `Добавить колонку после…` показывается только при праве `columns.create`.
- `Изменить название и тип…` показывается только при праве `columns.rename`.
- `Удалить колонку…` показывается только при праве `columns.delete`.

Если у пользователя есть только `columns.create`, меню содержит только один пункт: `Добавить колонку после…`.

Если у пользователя есть только `columns.rename`, меню содержит только один пункт: `Изменить название и тип…`.

Если у пользователя есть только `columns.delete`, меню содержит только один пункт: `Удалить колонку…` или его disabled-состояние для последней колонки.

## 7. Создание колонки из меню текущей колонки

### 7.1. Запуск сценария
Нажатие на пункт `Добавить колонку после…` открывает модальное окно создания колонки.

Используется тот же сценарий создания, что и раньше:
- заголовок модального окна: `Новая колонка`;
- обязательное поле названия;
- обязательное поле типа колонки;
- кнопки `Создать` и `Отмена`.

### 7.2. Начальные значения формы
Форма создания открывается со следующими начальными значениями:
- название пустое;
- placeholder названия: `Новая колонка`;
- тип колонки по умолчанию: `queue`.

Допустимые типы колонки остаются теми же, что уже поддерживаются на доске.

### 7.3. Обязательный текст-пояснение
В модальном окне над формой обязательно показывается пояснение:

`Новая колонка будет добавлена сразу после колонки "<название исходной колонки>".`

Используется именно название исходной колонки, актуальное на момент открытия модального окна.

### 7.4. Правило вставки
После успешного создания новая колонка вставляется строго на позицию сразу после исходной колонки в логическом порядке колонок.

Это означает:
- исходная колонка сохраняет своё место;
- новая колонка получает следующую позицию после исходной;
- все колонки, которые раньше шли после исходной, сдвигаются на одну позицию вправо в логическом порядке;
- относительный порядок остальных колонок сохраняется.

Добавление новой колонки в конец ряда допускается только тогда, когда исходная колонка уже была последней в логическом порядке.

### 7.5. Поведение после успеха
После успешного создания:
- модальное окно закрывается;
- доска остаётся на том же маршруте;
- список колонок обновляется без перехода на другой экран;
- горизонтальный ряд прокручивается так, чтобы новая колонка оказалась видимой пользователю;
- дополнительный автоматический переход в режим переименования новой колонки не выполняется.

### 7.6. Поведение при ошибке
Если создание завершилось ошибкой:
- модальное окно не закрывается;
- введённые пользователем значения сохраняются в форме;
- текст ошибки показывается в модальном окне.

## 8. Серверные правила создания колонки

### 8.1. Обязательный источник вставки
Сценарий создания колонки для экрана доски обязан выполняться с указанием исходной колонки.

Если исходная колонка не передана или не принадлежит текущей доске, операция должна завершаться ошибкой:

`Колонка не найдена на этой доске.`

### 8.2. Валидация полей
Перед созданием сохраняются действующие проверки:
- название обязательно;
- название не длиннее 50 символов;
- тип колонки обязан быть одним из разрешённых значений;
- пользователь обязан быть авторизован;
- у пользователя должно быть право `columns.create`.

### 8.3. Правило расчёта порядка
Порядок вставки рассчитывается по актуальному состоянию колонок в базе на момент выполнения серверной операции, а не по устаревшему клиентскому снимку.

Если за время между открытием модального окна и отправкой формы порядок колонок изменился, новая колонка всё равно вставляется сразу после той позиции, которую исходная колонка занимает в базе в момент создания.

Если исходная колонка была удалена до отправки формы, операция завершается ошибкой:

`Колонка не найдена на этой доске.`

### 8.4. Итоговый порядок после вставки
После успешной серверной операции набор позиций колонок на доске должен оставаться линейным и однозначным.

Ожидаемое состояние:
- каждая колонка принадлежит только одной позиции;
- порядок колонок соответствует UI;
- новая колонка находится строго после исходной колонки.

## 9. Удаление колонки и запрет на удаление последней

### 9.1. Базовое правило
Если на доске осталась ровно одна колонка, удалить её нельзя ни при каких условиях.

Это правило обязательно даже тогда, когда:
- колонка пустая;
- у пользователя есть право `columns.delete`;
- удаление выполняется не из UI, а прямым вызовом серверной операции.

### 9.2. Поведение меню для последней колонки
Если пользователь имеет право `columns.delete`, но колонка является последней на доске:
- пункт `Удалить колонку…` остаётся видимым в меню;
- пункт отображается в disabled-состоянии;
- под disabled-пунктом показывается неинтерактивное пояснение:

`Последнюю колонку на доске удалить нельзя.`

Модальное окно подтверждения удаления в этом состоянии не открывается.

### 9.3. Поведение меню для не последней колонки
Если на доске больше одной колонки и у пользователя есть право `columns.delete`, пункт `Удалить колонку…` работает как обычное действие удаления.

### 9.4. Правило при наличии карточек
Для колонок, которые не являются последними на доске, сохраняется действующее ограничение:
- колонку с карточками удалить нельзя;
- пользователь получает существующее понятное сообщение об ошибке про необходимость сначала перенести или удалить карточки.

### 9.5. Приоритет проверок на сервере
Серверная операция удаления должна проверять условия в таком порядке:

1. Пользователь авторизован.
2. Колонка существует и принадлежит указанной доске.
3. На доске больше одной колонки.
4. Выполняются остальные ограничения удаления, включая запрет удаления колонки с карточками.

Если нарушено правило пункта 3, операция завершается ошибкой:

`Нельзя удалить последнюю колонку на доске.`

Это сообщение имеет приоритет над сообщением о наличии карточек.

## 10. Матрица прав доступа

### 10.1. `columns.create`
Право `columns.create` даёт доступ только к действию `Добавить колонку после…`.

Наличие этого права не даёт автоматически права:
- менять название колонки;
- менять тип колонки;
- удалять колонку.

### 10.2. `columns.rename`
Право `columns.rename` даёт доступ только к действию `Изменить название и тип…`.

### 10.3. `columns.delete`
Право `columns.delete` даёт доступ к действию удаления, но не отменяет запрет на удаление последней колонки и не отменяет запрет на удаление колонки с карточками.

### 10.4. Независимость прав
Права `columns.create`, `columns.rename` и `columns.delete` трактуются независимо друг от друга.

UI не должен скрытно требовать наличие одного права для показа действия, которое относится к другому праву.

## 11. Что не должно меняться
Эта доработка не должна менять:
- существующий набор типов колонок;
- логику редактирования типа колонки;
- drag-and-drop колонок;
- drag-and-drop карточек;
- права на создание карточек;
- правила удаления карточек;
- порядок колонок при обычном drag-and-drop;
- структуру экрана доски вне переноса точки входа создания колонки.

## 12. Недопустимое поведение
Считаются ошибкой следующие состояния:
- на экране доски одновременно доступны и глобальная кнопка `+ Колонка`, и создание через меню колонки;
- новая колонка создаётся не после исходной, а в конце ряда при наличии колонок справа;
- новая колонка создаётся до исходной колонки;
- после создания нарушается относительный порядок остальных колонок;
- пункт `Добавить колонку после…` виден без права `columns.create`;
- пункт `Изменить название и тип…` виден без права `columns.rename`;
- пункт `Удалить колонку…` виден без права `columns.delete`;
- последнюю колонку можно удалить через UI;
- последнюю колонку можно удалить прямым серверным вызовом;
- для последней колонки открывается модальное окно подтверждения удаления;
- для последней колонки нет явного объяснения, почему удаление недоступно;
- удаление не последней пустой колонки блокируется только из-за наличия нового правила про последнюю колонку.

## 13. Критерии приёмки
Задача считается выполненной, если одновременно соблюдаются все условия:

1. На экране доски отсутствует отдельная кнопка `+ Колонка`.
2. Создание колонки запускается из меню конкретной колонки.
3. Пункт создания называется `Добавить колонку после…`.
4. После создания новая колонка появляется строго после колонки-источника.
5. Если колонка-источник была последней, новая колонка становится новой последней колонкой.
6. Если справа от исходной колонки уже были другие колонки, они сдвигаются вправо без нарушения своего взаимного порядка.
7. Пользователь с одним только правом `columns.create` может открыть меню и создать колонку, но не видит действий переименования и удаления.
8. Пользователь с одним только правом `columns.rename` видит только действие изменения названия и типа.
9. Пользователь с одним только правом `columns.delete` видит только действие удаления или его disabled-состояние для последней колонки.
10. Для последней колонки действие удаления недоступно в UI и сопровождается явным пояснением.
11. Сервер возвращает ошибку `Нельзя удалить последнюю колонку на доске.`, если попытаться удалить последнюю колонку любым способом.
12. Для не последней колонки сохраняется действующее ограничение на удаление при наличии карточек.

## 14. Статус вопросов
Открытых вопросов для этой спецификации нет.
