# План реализации "Плавающий drag-and-drop карточки" (для AI-агента)

Основано на `.ai/card-floating-dnd-specification.md`.

Цель плана: дать AI-агенту исполняемую декомпозицию для перевода drag-and-drop карточек к модели "свободного полёта" как в Trello: активная карточка уходит в overlay поверх доски, её размер фиксируется на старте drag, в списках колонок живёт ровно один slot вставки, а карточки в колонках перестраиваются в реальном времени ещё до drop.

## 0) Зафиксированные требования спецификации
- Карточка при фактическом начале drag немедленно перестаёт быть видимым элементом списка своей колонки.
- Активная карточка рендерится в отдельном верхнем слое и не обрезается колонкой, доской или scroll-контейнером.
- Во время drag не должно быть двух видимых экземпляров одной и той же карточки.
- Ширина и высота плавающей карточки фиксируются в момент старта drag и не меняются до завершения drag.
- Во время drag на доске существует ровно один slot вставки.
- Slot вставки всегда показывает актуальную будущую позицию drop.
- Высота slot вставки равна зафиксированной высоте перетаскиваемой карточки.
- Карточки в колонках обязаны перестраиваться прямо во время drag, а не только после drop.
- Позиция slot вставки определяется по текущей колонке наведения и вертикальной позиции указателя внутри неё.
- Если указатель временно ушёл с допустимой колонки, slot остаётся в последней валидной позиции drag-сеанса.
- При переносе между колонками не должно быть промежуточного "возврата" карточки в исходную колонку.
- После drop карточка должна оказаться ровно там, где slot был показан непосредственно перед отпусканием.
- После cancel карточка возвращается в исходную позицию, overlay и slot исчезают, модалка не открывается.
- Автоскролл колонок/доски не добавляется.
- Матрица прав и правила full-surface drag из `.ai/done/card-full-surface-dnd-agent-plan.md` сохраняются без расширения.

## 1) Текущий технический контекст проекта

### 1.1. Что уже есть в коде
- `web/src/app/boards/[boardId]/board-columns-dnd.tsx`
  - основной центр логики карточек и колонок;
  - `SortableBoardCard` использует `useSortable({ id: card.id })`;
  - `BoardCardRow` уже является общим визуальным телом карточки;
  - есть full-surface drag activator, anti-click после drag, touch sensor, матрица `move x open`;
  - `PointerSensor` настроен как `distance: 8`;
  - `TouchSensor` настроен как `delay: 250`, `tolerance: 8`;
  - `DndContext` уже использует `closestCorners`;
  - есть `onDragStart`, `onDragCancel`, `onDragEnd`;
  - локальный optimistic state хранит `columnItems`, `cardsById`, `cardOrderByColumn`.
- `web/src/app/boards/[boardId]/page.tsx`
  - источник `canMoveCards = has("cards.move")`;
  - источник full snapshot доски (`get_board_snapshot`);
  - сюда уже сведены карточки, колонки, preview items и права.
- `web/src/app/boards/[boardId]/column-types.ts`
  - источник логики `canOpenCardModal(...)`, `canEditCardContent(...)`, `canDeleteCard(...)`.
- `web/src/app/boards/[boardId]/actions.ts`
  - уже есть `reorderBoardCardsAction(...)`, который сохраняет итоговый layout карточек.

### 1.2. Что в текущей реализации уже полезно для этой задачи
- Drag карточек уже работает на full-surface карточки.
- Есть единый visual component карточки (`BoardCardRow`), который можно переиспользовать и в overlay.
- Уже есть защита от ложного открытия модалки после drag/cancel.
- Есть локальный optimistic layout и realtime/polling механика.
- Есть статический fallback-рендер до mount DnD и без DnD-прав.

### 1.3. Что сейчас прямо расходится со спецификацией
- `DragOverlay` или другой отдельный floating-layer сейчас не используется.
- Активная карточка остаётся тем же sortable-элементом списка с `opacity`/`transform`, а не отдельным "летящим" объектом.
- Для карточки есть workaround `transform.x = 0`, что является признаком drag "внутри колонки", а не свободного overlay.
- В списке нет полноценного slot вставки размера карточки.
- `EmptyColumnDrop` покрывает только пустую колонку и не реализует единый slot для всех сценариев.
- Будущая позиция карточки вычисляется только в `onDragEnd`, а не во время drag.
- `resolveCardDropTarget(...)` сейчас не реализует правило "верхняя половина / нижняя половина карточки".
- Межколоночное live-расталкивание до drop отсутствует как явная модель.
- Нет отдельной фиксации размеров карточки на старте drag.
- Нет отдельного drag-session state с `last valid slot`.

## 2) Рабочие правила для AI-агента
- Не менять продуктовую матрицу прав, уже зафиксированную в full-surface DnD.
- Не добавлять новый keyboard drag-flow.
- Не менять серверную бизнес-логику сохранения порядка шире, чем нужно для точной передачи итогового layout.
- Не добавлять новый автоскролл доски/колонок.
- Не менять состав preview карточки в overlay относительно карточки в колонке.
- Не трогать drag-and-drop колонок, кроме совместимости с новым card-drag flow.
- Не оставлять параллельно две модели активной карточки: либо overlay, либо старый inline-drag, но не обе одновременно.
- Любая спорная точка по `dnd-kit`, где придётся менять фундамент collision/сенсоры сразу и для карточек, и для колонок, является стоп-условием: сначала спросить пользователя.

## 3) Стратегия выполнения
1. Сначала зафиксировать новый drag-session contract: что считается активной карточкой, где хранится slot и как скрывается исходная карточка.
2. Затем отделить визуальное представление активной карточки от списка: ввести overlay и убрать старую модель "та же карточка двигается внутри списка".
3. После этого ввести единую модель slot вставки и live-перестройки layout во время drag.
4. Потом связать drop/cancel с этой новой моделью так, чтобы финальный persist использовал именно последний показанный slot.
5. Отдельно выровнять realtime, fallback и визуальные детали, чтобы overlay не конфликтовал с текущими optimistic/realtime механизмами.
6. В конце пройти ручную приёмку по всем acceptance criteria и только затем решать, нужны ли точечные тесты.

Критический порядок:
- Нельзя начинать косметическую полировку shadow/opacity/animation, пока не доказано, что карточка реально вынута из списка, slot живёт отдельно и layout перестраивается прямо во время drag.

## 6) Предпочтительные технические решения

### 6.1. Overlay
- Предпочтительно использовать `DragOverlay`.
- В overlay рендерить тот же `BoardCardRow`, но в режиме:
  - без `listeners/attributes`;
  - без открытия модалки;
  - с зафиксированным `width/height`.

### 6.2. Источник истины по layout
- Реальный порядок карточек в `local.cardOrderByColumn` должен оставаться порядком реальных карточек.
- Drag-time slot лучше держать отдельно как `dragSession.currentSlot`.
- Для рендера строить производный `displayCardFlow`, который:
  - исключает `activeCardId` из source column;
  - вставляет `slot` в нужное место;
  - не притворяется, что карточка уже реально сохранена в БД.

### 6.3. Вычисление позиции вставки
- Не полагаться только на `closestCorners` как на финальный источник индекса.
- Использовать DOM-геометрию карточек/контейнеров для определения:
  - текущей колонки;
  - верхней/нижней половины карточки;
  - конца списка;
  - пустой колонки.

### 6.4. Persist drop
- В `reorderBoardCardsAction(...)` отправлять layout, построенный из реального порядка карточек после применения финального slot.
- Slot в persist не отправляется как сущность, а только материализуется в итоговый порядок `card_ids`.

## 7) Риски и защита от ошибок
- **Риск: overlay будет рендериться поверх доски, но исходная карточка останется видимой в списке.**
  - Защита: отдельная проверка "one visible instance" до любой визуальной полировки.
- **Риск: slot начнёт жить как вторая "псевдокарточка", а реальный layout станет неочевидным.**
  - Защита: разделить real layout и drag projection, не смешивать их в одном массиве без явной модели.
- **Риск: `onDragOver` + realtime/polling будут перетирать друг друга.**
  - Защита: на время drag изолировать входящие layout-update или безопасно откладывать их.
- **Риск: drop при `over = null` приведёт к потере предсказуемости.**
  - Защита: хранить `lastValidSlot` как явную часть drag-session state.
- **Риск: overlay потребует большого рефакторинга `BoardCardRow`.**
  - Защита: переиспользовать существующий компонент и добавлять только минимальные props для overlay-mode.
- **Риск: колонка и карточка начнут конфликтовать в одном `DndContext`.**
  - Защита: не менять column DnD шире необходимого; card-flow дорабатывать точечно.

## 8) Стоп-условия: когда агент обязан остановиться и спросить пользователя
- Если для корректного overlay потребуется продуктово значимое изменение поведения column DnD, а не локальная совместимость.
- Если окажется, что `dnd-kit` в текущей архитектуре не даёт надёжно реализовать правило "верхняя половина / нижняя половина" без крупной смены collision strategy для всей доски.
- Если без нового автоскролла невозможно добиться приемлемого поведения, и это становится уже продуктовым компромиссом, а не локальным дефектом.
- Если реализация потребует менять серверный контракт перестановки карточек, а не только клиентскую визуально-поведенческую модель.

## 9) Критерий завершения всей задачи
Задача считается завершённой только когда одновременно выполнено всё ниже:
- активная карточка во время drag рендерится как отдельный floating overlay;
- её размер фиксируется на старте drag и не меняется до конца drag-сеанса;
- в колонках существует ровно один slot вставки;
- исходная карточка как элемент списка на время drag не видна;
- карточки в колонках перестраиваются прямо во время drag;
- межколоночный переход происходит без визуального возврата карточки в source column;
- drop фиксирует карточку ровно в позиции последнего показанного slot;
- cancel возвращает карточку в исходное место без открытия модалки;
- без права `cards.move` overlay/slot/live-reorder вообще не возникают;
- изменённые файлы проходят lint/typecheck и ручную приёмку по спецификации.
