## Назначение
Этот документ предназначен для AI-агента, который будет внедрять требования из `.ai/board-screen-fit-specification.md`.

Цель:
- убрать вертикальный и горизонтальный scroll страницы на маршруте доски;
- разрешить горизонтальный scroll только у ряда колонок;
- оставить единственные разрешённые внутренние scroll-области у ряда колонок по оси `x` и у списка карточек внутри видимой колонки по оси `y`;
- не разделять колонки на экраны/страницы;
- зафиксировать высотную и ширинную модель board-screen без двусмысленности.

## Главные инварианты реализации
- Нельзя допускать scroll у документа, `body` и route-root доски.
- Нельзя допускать горизонтальный scroll нигде, кроме контейнера ряда колонок.
- На desktop/tablet частично видимая следующая колонка на границе viewport допустима и ожидаема при горизонтальном scroll.
- На mobile в viewport должна помещаться одна полная колонка; переход к следующим выполняется горизонтальным scroll ряда.
- Нельзя менять бизнес-логику карточек, прав доступа, модалок и содержимого карточек вне того, что требуется для layout и горизонтального ряда колонок.
- Нельзя вводить новый способ перемещения карточек, не связанный с текущей DnD-моделью доски.
- Нельзя менять глобальный reorder колонок так, чтобы ломался ожидаемый порядок единого горизонтального ряда.
- Если для реализации требуется решение, которого нет в спецификации, агент обязан остановиться и спросить пользователя.

## Целевое архитектурное решение

### 1. Высотный shell board-route
- После глобального header маршрут доски занимает весь остаток viewport.
- У board-screen ровно 3 вертикальные зоны:
  1. верхняя строка доски;
  2. строка управления доской;
  3. полотно колонок.
- Верхние 2 зоны имеют фиксированную высоту по breakpoint-правилам спецификации.
- Полотно колонок получает остаток высоты через CSS (`minmax(0,1fr)` / `flex-1 min-h-0`) без JS-измерений, если это возможно.

### 2. Модель колонок в полотне
- В DOM и UI одновременно рендерятся все колонки в едином горизонтальном ряду.
- Если ширины viewport не хватает, используется горизонтальный scroll контейнера ряда колонок.
- Для mobile в viewport должна помещаться одна полная колонка, а переход к следующим выполняется горизонтальным scroll ряда.
- Переключатель экранов/страниц колонок не используется.

### 3. Модель одной колонки
- Колонка занимает полную высоту полотна колонок.
- Колонка состоит из:
  1. заголовка;
  2. области списка карточек;
  3. нижней зоны с кнопкой создания карточки либо пустым footer-space.
- Scroll разрешён только у middle-section со списком карточек.
- Header и footer колонки не участвуют в scroll.

### 4. Модель DnD
- DnD колонок работает в пределах единого горизонтального ряда всех колонок.
- DnD карточек работает между колонками текущей доски без экранной сегментации.
- Отдельные действия меню для переноса колонки между экранами не требуются.

#### Результат аудита T01
- Текущий global app-shell в `web/src/app/layout.tsx` устроен как `div.flex.h-dvh.min-h-0.flex-col.overflow-hidden.pt-2`, но дочерний `main.mx-auto ... flex-1 overflow-y-auto px-4` остаётся вертикальным scroll-owner для всех маршрутов, включая доску. Это прямо конфликтует с целевым правилом, по которому board-route после header должен занимать весь остаток viewport без собственного page-scroll.
- `body` в `web/src/app/layout.tsx` и `web/src/app/globals.css` не объявлен scroll-owner явно, но текущий `main` уже создаёт отдельную scroll-цепочку маршрута; значит, проблему нужно устранять на уровне layout-shell, а не только внутри доски.
- `web/src/app/boards/[boardId]/page.tsx` уже ограничивает внутреннюю область через `grid h-full min-h-0 grid-rows-[auto_minmax(0,1fr)]`, однако верхняя строка собрана с `flex-wrap`, а сама страница использует `gap-4 px-4`, то есть высоты и внутренние отступы ещё не приведены к фиксированной модели `56/48/40` и `16/12`.
- `web/src/app/boards/[boardId]/board-canvas.tsx` сейчас делает scroll-owner из canvas-viewport: `div.min-h-0.flex-1.overflow-auto`. Именно этот узел одновременно разрешает вертикальный и горизонтальный scroll содержимого доски, что противоречит целевой модели "scroll только внутри card-list".
- `web/src/app/boards/[boardId]/board-columns-dnd.tsx` рендерит ряд колонок как `flex w-max min-w-full gap-4 pb-2` и в SSR, и в hydrated DnD-ветке. Из-за `w-max` весь набор колонок растягивает внутреннюю ширину за пределы viewport, а `overflow-auto` родителя из `BoardCanvas` превращает это в горизонтальный scroll полотна.
- Видимые колонки сейчас имеют фиксированную ширину `w-72`, но не получают `h-full`; список карточек собран как обычный `flex-col gap-2` без собственного `overflow-y-auto`, поэтому вертикальное переполнение уходит наружу в canvas-scroll, а не изолируется внутри колонки.
- `web/src/app/boards/[boardId]/board-background-frame.tsx` выступает только визуальной оболочкой (`section.relative` + абсолютный background-layer + `div.relative.z-10.flex.h-full.min-h-0.flex-1.flex-col`) и на текущем шаге сам scroll не создаёт; его можно сохранять как фоновой слой, пока scroll убирается из layout/canvas/grid.
- Текущий список узлов, которые должны перестать быть scroll-owner на пути к целевой модели:
  - глобальный `main` в `web/src/app/layout.tsx`;
  - canvas viewport `div.min-h-0.flex-1.overflow-auto` в `web/src/app/boards/[boardId]/board-canvas.tsx`;
  - горизонтальный row контейнер колонок `div.flex.w-max.min-w-full...` в `web/src/app/boards/[boardId]/board-columns-dnd.tsx`.
- Текущий список узлов, которые должны стать/остаться не-scroll контейнерами с жёстким `min-h-0`:
  - route root доски в `web/src/app/boards/[boardId]/page.tsx`;
  - `BoardBackgroundFrame` как визуальный shell;
  - будущий columns-canvas/page-shell.
- Предварительный вывод для следующего шага: `BoardCanvas` логично оставлять точкой входа board-shell, но его нужно разрезать на фиксированную control-row и отдельную columns/grid-область без `overflow-auto` у всего полотна.

### Фаза 2. Заморозка целевой DOM-модели board-screen
- Определить, какой компонент отвечает за:
  - board top row;
  - board control row;

## Исполняемый порядок для AI-агента
1. Сначала закрыть `T01` и `T02`, не меняя поведение по частям до фиксации целевой модели.
2. Затем собрать высотный shell (`T03`), после чего отдельно стабилизировать верхнюю строку (`T04`) и строку управления (`T05`).
3. До любых DnD-правок зафиксировать геометрию и модель горизонтального ряда (`T06`-`T08`).
4. После этого перевести рендер на единый ряд колонок без экранной нарезки (`T09`-`T10`).
5. Только затем перестраивать shell колонки и card-list scroll (`T11`-`T12`).
6. DnD-ограничения и UX-очистку от экранной логики выполнять на финальном layout (`T13`-`T15`).
7. В конце отдельно закрыть overflow/фон/SSR/resize (`T16`-`T17`) и только потом линтеры и ручную проверку (`T18`).

## Чек-лист приёмки по спецификации
- На `/boards/{boardId}` отсутствует vertical page-scroll.
- На `/boards/{boardId}` отсутствует horizontal page-scroll на уровне документа/route-root.
- После global header board-screen занимает весь остаток viewport.
- Верхняя строка доски имеет фиксированную высоту 56 px на desktop/tablet и 48 px на mobile.
- Строка управления доской имеет фиксированную высоту 40 px на всех ширинах.
- Внутренние отступы доски соответствуют 16 px на desktop/tablet и 12 px на mobile.
- Полотно колонок занимает весь оставшийся остаток высоты.
- На desktop/tablet ширина видимой колонки равна 288 px.
- На mobile в viewport помещается одна полная колонка.
- Горизонтальный scroll разрешён только у ряда колонок внутри полотна.
- Каждая видимая колонка занимает полную высоту полотна.
- Scroll карточек существует только внутри списка карточек конкретной видимой колонки.
- Заголовок колонки не двигается при scroll карточек.
- Кнопка добавления карточки не двигается при scroll карточек.
- При избытке колонок работает горизонтальный scroll ряда колонок.
- Создание/удаление колонок не требует экранной навигации.
- DnD колонок работает в едином горизонтальном ряду.
- DnD карточек работает между колонками доски без экранной сегментации.

## Правило остановки
Агент обязан остановиться и спросить пользователя, если:
- для расчёта ширины полотна нельзя однозначно выбрать между CSS-only и `ResizeObserver`, и обе реализации дают разные UX-эффекты;
- серверная модель порядка колонок не позволяет без неоднозначности поддерживать стабильный глобальный порядок единого горизонтального ряда;
- текущий layout глобального `main` в `layout.tsx` делает невозможным выполнение спецификации без рискованого изменения глобального app-shell;
- обнаружится скрытая зависимость, из-за которой текущий UX или серверная логика фактически завязаны на экранную сегментацию колонок;
- для соблюдения спецификации потребуется UX, не описанный в `.ai/board-screen-fit-specification.md`.
