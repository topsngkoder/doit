# План реализации: card modal layout

## Назначение
Этот документ предназначен для AI-агента, который будет внедрять требования из `.ai/card-modal-layout-specification.md`.

Цель: привести модалку карточки к точным требованиям по размеру, позиционированию, двухколоночной раскладке и прокрутке без регрессий для остальных модалок проекта.

## Источник требований
- Основная спецификация: `.ai/card-modal-layout-specification.md`
- Открытых вопросов в спецификации нет.

## Целевые файлы
- `web/src/app/boards/[boardId]/edit-card-modal.tsx`
- `web/src/app/boards/[boardId]/card-comments-sidebar.tsx`
- `web/src/components/ui/modal.tsx`

## Связанные файлы для регрессионной проверки
- `web/src/app/boards/[boardId]/create-card-modal.tsx`
- `web/src/app/boards/[boardId]/board-card-preview-button.tsx`
- все остальные места использования `Modal`

## Ограничения
- Не менять бизнес-логику карточки, комментариев, прав доступа и состав полей.
- Не менять тексты, если это не требуется для layout.
- Не делать глобальную поломку базового `Modal`.
- Если в процессе выяснится, что спецификацию нельзя реализовать без неоднозначного решения, остановиться и спросить пользователя.

## Ключевые выводы по текущему состоянию
- Базовый `Modal` центрируется через `items-center justify-center`, имеет `max-h-[90vh]`, `max-w-lg`, общий `overflow-hidden`, и не умеет управлять вертикальным смещением по viewport.
- `edit-card-modal.tsx` уже содержит двухзонную структуру, но:
  - размеры модалки заданы приблизительно, а не по спецификации;
  - раскладка включается только на `xl`, а нужна по явным брейкпоинтам `desktop/tablet/mobile`;
  - правая колонка имеет фиксированную ширину `27rem/30rem`, а нужна пропорция `2/3 : 1/3`;
  - mobile/tablet/desktop поведение не соответствует точным значениям viewport;
  - текущая прокрутка и высоты завязаны на `min-h`/`max-h`, а не на строгую модель со скроллом внутри колонок.
- `card-comments-sidebar.tsx` уже умеет отдельно прокручивать список комментариев, но его нужно встроить в новую модель высот без конкурирующего внешнего скролла.

## Стратегия реализации
1. Не ломать существующий `Modal` для других сценариев.
2. Добавить в `Modal` опциональные возможности для card modal:
   - кастомизация контейнера/оверлея;
   - кастомизация панели диалога;
   - возможность отключить дефолтное вертикальное центрирование;
   - возможность задать специфичный layout body без навязанного `overflow-y-auto`.
3. Реализовать layout card modal как отдельную конфигурацию поверх общего `Modal`, а не как глобальное изменение всех диалогов.
4. Зафиксировать брейкпоинты по спецификации:
   - `desktop >= 1280`
   - `tablet 768..1279`
   - `mobile <= 767`
5. Для `desktop` и `tablet` построить ровно две колонки с вертикальным разделителем и независимыми скроллами.
6. Для `mobile` отключить двухколоночность и перейти на единый scroll-контейнер.

## Детальная декомпозиция

### Фаза 1. Подготовка
- Проверить все текущие использования `Modal`, чтобы понять, какие новые пропсы можно добавить без регрессий.
- Выбрать способ расширения `Modal`:
  - либо новые пропсы вида `overlayClassName`, `panelClassName`, `viewportMode`;
  - либо нейтральные опциональные пропсы для точечной настройки card modal.
- Не менять дефолтное поведение `Modal` для существующих диалогов.

### Фаза 2. Базовый modal shell
- Обновить `web/src/components/ui/modal.tsx`.
- Добавить опциональную настройку overlay/panel/body, достаточную для:
  - desktop: `70vw x 70vh`, центр по X, центр панели на `46vh` по Y;
  - tablet: `92vw x 82vh`, центр панели на `48vh` по Y;
  - mobile: `96vw x 90vh`, обычное центрирование без вертикального смещения.
- Сохранить текущее API совместимым для остальных вызовов.
- Убедиться, что внешний контейнер модалки не создает дополнительный scroll поверх внутренних секций card modal.

### Фаза 3. Layout card modal
- Обновить `web/src/app/boards/[boardId]/edit-card-modal.tsx`.
- Вынести конфигурацию размеров/позиционирования в классы, завязанные на нужные брейкпоинты.
- Сохранить header на всю ширину модалки:
  - название карточки;
  - кнопка закрытия;
  - без деления заголовка на колонки.
- Перестроить body:
  - `desktop/tablet`: две колонки;
  - `mobile`: один столбец.
- В `desktop/tablet` задать пропорцию строго `66.6667% / 33.3333%`.
- Заменить текущую фиксированную ширину comments sidebar на пропорциональную колонку.
- Добавить явный вертикальный разделитель между левой и правой колонками только для `desktop/tablet`.

### Фаза 4. Левая колонка
- Оставить в левой колонке только:
  - вкладки `Детали` / `История`;
  - содержимое активной вкладки;
  - поля карточки;
  - действия сохранения и удаления.
- Убедиться, что вкладка `История` не уезжает вправо и не становится отдельной колонкой.
- Настроить левую колонку как независимый вертикальный scroll-контейнер в `desktop/tablet`.

### Фаза 5. Правая колонка
- Оставить в правой колонке только блок комментариев.
- Проверить, что внутри `CardCommentsSidebar` нет элементов, которые нарушают идею "только комментарии".
- Настроить правую колонку как независимый вертикальный scroll-контейнер в `desktop/tablet`.
- На `mobile` перестроить блок комментариев ниже карточки, без отдельной колоночной прокрутки.

### Фаза 6. Scroll model
- Для `desktop/tablet` реализовать строгую схему:
  - модалка имеет фиксированную высоту по viewport;
  - header занимает верхнюю область;
  - content area занимает оставшуюся высоту;
  - обе колонки получают одинаковую доступную высоту;
  - каждая колонка скроллится сама;
  - внешний контейнер не создает второй конкурирующий вертикальный scroll.
- Для `mobile` реализовать:
  - один поток контента;
  - одна общая вертикальная прокрутка модалки;
  - отсутствие независимых скроллов у бывших колонок.

### Фаза 7. Проверка адаптивности
- Проверить вручную три режима:
  - `>=1280`
  - `768..1279`
  - `<=767`
- Отдельно проверить переходы на граничных значениях:
  - `767`
  - `768`
  - `1279`
  - `1280`
- Проверить длинный контент:
  - длинная история;
  - длинный список комментариев;
  - длинное описание;
  - много пользовательских полей.

### Фаза 8. Регрессионная проверка
- Проверить, что остальные модалки по-прежнему открываются и выглядят как раньше:
  - `create-card-modal`
  - `board-card-preview-button`
  - минимум 1-2 другие модалки на доске
- Проверить закрытие по overlay и кнопке закрытия.
- Проверить, что body scroll страницы не начинает конкурировать с modal scroll при открытой карточке.

1. Выполнить `T01-T02`.

3. Выполнить `T03`.
4. Сразу после `T03` быстро проверить, что простые модалки все еще открываются.
5. Выполнить `T04-T10`.

   - desktop;
   - tablet;
   - mobile.
7. Выполнить `T11-T13`.
8. В финальном отчете перечислить:
   - какие файлы изменены;
   - как реализованы брейкпоинты;
   - как решена прокрутка;
   - какие сценарии проверены вручную;
   - какие риски остались.

## Правило остановки
Остановиться и спросить пользователя, если в ходе реализации всплывет хотя бы одно из условий:
- требуется изменить поведение общего `Modal` так, что могут измениться все остальные диалоги;
- найдено противоречие между фактической DOM-структурой и обязательной спецификацией;
- реализация упирается в внешний layout/portal/scroll-lock механизм, который не виден в текущих файлах;
- для точного выполнения спецификации нужен новый UX-компромисс, не описанный в документе.
