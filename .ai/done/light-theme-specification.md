# Спецификация: светлая тема приложения

## 1. Назначение документа
Документ фиксирует конечные и обязательные требования к светлой теме приложения `Doit`.

Цель доработки:
- добавить в приложение полноценную светлую тему;
- сохранить текущую тёмную тему как альтернативный режим;
- зафиксировать точные визуальные решения без двусмысленности;
- исключить появление локальных “самодельных” цветовых решений в разных компонентах.

В этой спецификации нет кода. Все решения ниже являются обязательными.

## 2. Объект доработки
Светлая тема распространяется на весь пользовательский интерфейс веб-приложения:
- публичный лендинг;
- страницы входа и регистрации;
- глобальный header;
- страницы досок, профиля и уведомлений;
- все модалки;
- dropdown/popover/контекстные меню;
- формы;
- toast-уведомления;
- интерфейс доски, включая колонки, карточки, превью карточек и модалку карточки.

Исключения, которые не перекрашиваются темой:
- пользовательский цвет фона доски;
- пользовательское изображение фона доски;
- пользовательские цвета меток;
- пользовательские цвета опций select-полей.

## 3. Фиксированные продуктовые решения

### 3.1. Поддерживаемые темы
В приложении существуют ровно 2 темы:
1. `dark` — текущая тёмная тема;
2. `light` — новая светлая тема из этого документа.

Третьего режима нет.

Режима “Системная” нет.

### 3.2. Значение по умолчанию
Если пользователь ещё не выбирал тему явно, приложение открывается в теме `dark`.

Это решение сохраняет текущее поведение продукта и не меняет визуальный режим существующих пользователей без их действия.

### 3.3. Где пользователь переключает тему
Переключение темы размещается только на странице `Личный кабинет`.

Новый блок `Тема интерфейса` располагается:
- после заголовка страницы `Личный кабинет`;
- перед блоком аватара;
- перед формой профиля.

Внутри блока есть ровно 2 взаимоисключающих варианта:
1. `Тёмная`
2. `Светлая`

Toggle, checkbox, выпадающий список и отдельная страница настроек для темы не используются.

### 3.4. Как применяется выбор
После смены темы интерфейс должен переключаться сразу в текущей сессии.

Дополнительное подтверждение, отдельная кнопка `Сохранить` и перезагрузка страницы не требуются.

### 3.5. Хранение выбора
Выбор темы хранится локально в браузере.

Синхронизации темы между устройствами и через профиль пользователя в базе данных в рамках этой задачи нет.

### 3.6. Публичные страницы
Лендинг, вход и регистрация используют ту же тему, что была выбрана ранее в этом браузере.

Если пользователь тему не выбирал, эти страницы также открываются в `dark`.

## 4. Базовые правила темизации

### 4.1. Единая система токенов
Все цвета, границы, тени, фоны, состояния hover/focus/disabled и overlay должны определяться через единую семантическую систему токенов.

Недопустимо:
- задавать новые цвета произвольно внутри отдельных компонентов;
- оставлять жёстко прошитые тёмные значения в разметке компонентов;
- вводить отдельные “почти такие же” токены для однотипных элементов.

### 4.2. Обязательное покрытие токенами
Токены обязаны покрывать минимум следующие сущности:
- фон страницы;
- фон поверхности;
- фон приглушённой поверхности;
- фон поднятой поверхности;
- основной текст;
- вторичный текст;
- третичный текст;
- текст на тёмной или акцентной кнопке;
- стандартная граница;
- усиленная граница;
- акцентная граница;
- overlay модалки;
- focus ring;
- destructive/success/warning/info состояния.

### 4.3. Первая отрисовка
Выбранная тема должна применяться до первой видимой отрисовки страницы.

Белая вспышка перед включением тёмной темы и тёмная вспышка перед включением светлой темы считаются дефектом.

### 4.4. Browser color scheme
Для `light` обязателен `light color scheme`.

Для `dark` обязателен `dark color scheme`.

## 9. Доска: обязательные правила светлой темы

### 9.1. Фон доски
Если у доски выбран пользовательский цвет, этот цвет должен остаться точным и не заменяться темой.

Если у доски выбрано пользовательское изображение, изображение остаётся исходным.

### 9.2. Подложка поверх изображения доски
Чтобы рабочие элементы на фоне изображения оставались читаемыми, поверх фонового изображения обязательно накладывается светлая вуаль:
- `rgba(255, 255, 255, 0.18)`

Тёмная вуаль для светлой темы не используется.

### 9.3. Колонки доски
Колонка в светлой теме должна быть читаемой поверх любого допустимого фона доски.

Точные значения:
- фон колонки: `rgba(255, 255, 255, 0.88)`
- граница колонки: `rgba(184, 196, 211, 0.55)`
- тень колонки: `0 12px 28px rgba(15, 23, 42, 0.14)`
- заголовок колонки: `#0F172A`
- служебный текст в колонке: `#64748B`

### 9.4. Карточка в колонке
Карточка доски в светлой теме:
- фон `#FFFFFF`
- граница `#D7DFE8`
- hover граница `#94A3B8`
- основной текст `#0F172A`
- вторичный текст `#64748B`
- тень стандартной карточки

Карточка не должна сливаться с колонкой.

### 9.5. Кнопка создания карточки
Кнопка `+ Карточка` внизу колонки:
- фон `rgba(255, 255, 255, 0.62)`
- граница dashed `#94A3B8`
- текст `#334155`
- hover фон `rgba(255, 255, 255, 0.80)`
- hover граница `#2563EB`
- hover текст `#1D4ED8`

### 9.6. Цветные метки и цветные select-опции
Во всех местах светлой темы, где отображается пользовательская метка или цветная select-опция:
- граница чипа должна использовать исходный пользовательский цвет без изменения;
- фон чипа должен быть смесью `16%` пользовательского цвета и `84%` белой поверхности;
- текст внутри чипа должен быть `#0F172A`.

Это правило обязательно для:
- превью карточек на доске;
- селектора меток в модалке карточки;
- отображения select-полей;
- каталогов меток и опций.

### 9.7. Аватары участников
Обычный avatar chip:
- фон `#E2E8F0`
- текст `#334155`
- граница `#FFFFFF`

Аватар с выделением ответственного или автора, если такое состояние уже есть в текущем интерфейсе:
- использовать акцент `#D97706` для обводки;
- не использовать неоновые или тёмные glow-эффекты.

## 10. Модалка карточки в светлой теме

### 10.1. Общий принцип
Спецификация светлой темы не меняет layout модалки карточки, уже зафиксированный отдельным документом.

Светлая тема меняет только визуальный язык этой модалки.

### 10.2. Поверхности
В модалке карточки:
- фон panel: `#FFFFFF`
- фон левой колонки: `#FFFFFF`
- фон правой колонки комментариев: `#F8FAFC`
- границы и разделители: `#E5EAF1`

### 10.3. Заголовок и вкладки
Заголовок карточки:
- основной текст `#0F172A`
- вспомогательный текст `#64748B`

Вкладки `Детали` / `История`:
- неактивная вкладка: прозрачный фон, текст `#64748B`
- hover неактивной вкладки: фон `#EEF2F7`, текст `#334155`
- активная вкладка: фон `#EFF6FF`, текст `#1D4ED8`, граница `#BFDBFE`

### 10.4. Комментарии
Список комментариев и форма комментария в правой колонке используют:
- фон контейнера `#F8FAFC`
- фон вложенной карточки комментария `#FFFFFF`
- границу `#E5EAF1`
- основной текст `#0F172A`
- служебный текст `#64748B`

Ответ на комментарий, цитаты и вспомогательные блоки не должны использовать тёмные подложки.

## 11. Состояния и доступность

### 11.1. Focus
Каждый интерактивный элемент в светлой теме обязан иметь явно видимый focus state.

Фиксированное правило:
- толщина focus ring: `2 px`
- цвет focus ring: `#93C5FD`
- состояние должно быть заметно и на белом фоне, и на фоне изображения доски

### 11.2. Hover
Hover-состояния в светлой теме обязаны быть мягкими и читабельными.

Недопустимо:
- затемнять белую поверхность до грязно-серого состояния;
- делать hover настолько слабым, что он не читается;
- использовать тёмные hover-плашки из текущей тёмной темы.

### 11.3. Disabled
Disabled-состояние:
- визуально читается как отключённое;
- не исчезает полностью;
- сохраняет читаемость текста.

Базовое правило:
- непрозрачность `60%`
- курсор обычный, не pointer

### 11.4. Контраст
Минимальные требования:
- основной текст на основных поверхностях: не ниже `7:1`
- обычный текст в интерактивных контролах: не ниже `4.5:1`
- placeholder допускается слабее обычного текста, но должен оставаться различимым

## 12. Ограничения реализации

### 12.1. Нельзя опираться на вводящие в заблуждение имена цветов
В существующем проекте используются алиасы, где семантика имени цвета не совпадает с фактическим визуальным результатом.

В итоговой системе темы недопустимо опираться на такие названия как на источник дизайнерского смысла.

Обязательное правило:
- тема должна строиться от семантических токенов интерфейса;
- имя utility-класса не считается спецификацией цвета.

### 12.2. Нельзя оставлять жёстко тёмный root
Жёстко заданные тёмные значения на уровне `html`, `body`, inline-style и базовых обёрток должны быть приведены к теме.

Ситуация, при которой светлая тема включена, а root всё ещё остаётся тёмным, считается дефектом.

### 12.3. Нельзя менять layout ради темы
Светлая тема не должна:
- менять информационную архитектуру экранов;
- переставлять блоки;
- менять размеры модалок;
- менять брейкпоинты;
- менять тексты;
- менять бизнес-логику.

## 13. Критерии приёмки
Задача считается выполненной, если одновременно соблюдаются все условия:

1. В приложении доступны ровно две темы: `dark` и `light`.
2. При отсутствии пользовательского выбора приложение остаётся в `dark`.
3. На странице `Личный кабинет` есть отдельный блок `Тема интерфейса` с двумя взаимоисключающими вариантами: `Тёмная` и `Светлая`.
4. После выбора `Светлая` вся оболочка приложения, страницы, формы, модалки, dropdown, popover и toast переходят в светлую палитру без перезагрузки.
5. При следующем открытии приложения в том же браузере выбранная тема сохраняется.
6. В светлой теме не остаётся крупных тёмных поверхностей, кроме пользовательских фонов досок и пользовательских цветных меток/опций.
7. На доске колонки и карточки остаются читаемыми и визуально отделёнными как на цветном фоне, так и на фоне изображения.
8. Модалка карточки в светлой теме сохраняет текущий layout, но полностью переходит на светлые поверхности и светлые разделители.
9. Непрочитанные уведомления визуально выделяются светлой информационной подложкой, а не тёмной плашкой.
10. Focus-состояние заметно на всех интерактивных элементах.
11. При включении светлой темы нет тёмного root-фона и нет смешения светлой страницы с тёмным body.
12. При первой отрисовке не возникает вспышки неправильной темы.

## 14. Что не входит в задачу
- redesign логотипа;
- изменение шрифтов;
- изменение layout страниц;
- добавление режима `Системная`;
- синхронизация темы между устройствами;
- high-contrast theme;
- переработка пользовательских цветов меток;
- переработка пользовательского выбора фонового цвета или изображения доски;
- изменение бизнес-логики досок, карточек, комментариев, профиля и уведомлений.
