# Спецификация изменения: вся поверхность карточки как зона drag-and-drop

## 1. Назначение документа
Этот документ фиксирует конечные требования к поведению карточки на доске при перетаскивании.

Цель изменения:
- сделать зоной drag-and-drop не только ручку `⋮⋮`, а всю визуальную поверхность карточки;
- включить в drag-зону название карточки;
- убрать отдельную кликабельность внутренних элементов карточки;
- отложить любой специальный клик-функционал по полям карточки на отдельную будущую доработку.

Документ является целевой спецификацией для реализации и приёмки. В рамках этой спецификации не допускаются альтернативные трактовки поведения.

## 2. Область применения
Изменение применяется только к карточкам в колонках доски на основном экране доски.

В рамках этой спецификации:
- рассматривается только карточка в списке карточек колонки;
- не рассматривается содержимое открытой карточки в модальном окне;
- не рассматриваются отдельные сценарии кликов по меткам, адресу, ответственному, комментариям и другим полям;
- не рассматривается изменение состава элементов превью карточки.

## 3. Термины

### 3.1. Поверхность карточки
`Поверхность карточки` — это весь видимый прямоугольный блок карточки в колонке доски, включая:
- пустое пространство внутри карточки;
- название карточки;
- метки;
- блок участников;
- индикатор количества комментариев;
- блок ответственного;
- все пользовательские поля, показанные в превью карточки;
- любые визуальные чипы и текстовые элементы внутри карточки.

Исключений внутри поверхности карточки в этой доработке нет.

### 3.2. Внутренние элементы карточки
`Внутренние элементы карточки` — все отображаемые элементы внутри карточки, кроме внешнего контейнера карточки как единой зоны взаимодействия.

## 4. Зафиксированные решения

### 4.1. Общая зона перетаскивания
Вся поверхность карточки является единой зоной drag-and-drop карточки.

Следствия:
- пользователь может начать перетаскивание из любой точки карточки;
- название карточки входит в drag-зону;
- внутренние элементы карточки не создают отдельных “запрещённых” участков;
- отдельная ручка `⋮⋮` больше не является обязательной зоной старта drag-and-drop.

### 4.2. Судьба ручки `⋮⋮`
В этой доработке визуальная ручка `⋮⋮` убирается из интерфейса карточки полностью.

После изменения в карточке не должно оставаться отдельного визуального элемента, который обозначает drag-зону как особую область внутри карточки.

### 4.3. Механика начала drag-and-drop
Механика старта перетаскивания должна остаться такой же, как у текущей ручки `⋮⋮`.

Это означает:
- используется тот же принцип распознавания начала перетаскивания;
- используется тот же порог перехода из обычного нажатия в drag;
- используется тот же сценарий завершения, отмены и drop;
- меняется только область-активатор: вместо ручки используется вся поверхность карточки.

### 4.4. Поведение обычного клика по карточке
Если пользователь имеет право открыть карточку и drag не начался, обычный клик или тап по любой точке поверхности карточки открывает карточку так же, как это работает сейчас по клику на саму карточку.

Если в процессе взаимодействия начался drag, открытие карточки не происходит.

После завершения drag или после его отмены карточка не должна открываться автоматически.

### 4.5. Поведение внутренних элементов карточки
Ни один внутренний элемент карточки не имеет собственного кликового поведения в рамках этой доработки.

Это правило распространяется на все внутренние элементы без исключений, включая:
- метки;
- адрес;
- ссылки;
- даты;
- select-поля;
- текстовые пользовательские поля;
- ответственного;
- участников;
- счётчик комментариев;
- любые другие элементы превью карточки, которые отображаются внутри карточки.

Итоговое правило:
- внутренний элемент не должен выполнять собственное действие по клику;
- внутренний элемент не должен открывать отдельный экран, ссылку, поповер, меню или редактор;
- внутренний элемент не должен перехватывать drag-and-drop;
- взаимодействие с любой точкой внутри карточки подчиняется только общему правилу карточки: либо начинается drag, либо выполняется обычное действие самой карточки.

### 4.6. Некликабельность полей карточки
Фраза “поля карточки некликабельны” в рамках этой спецификации означает следующее:
- у поля нет собственного пользовательского действия;
- поле нельзя использовать как отдельную ссылку;
- поле нельзя использовать как отдельную кнопку;
- поле нельзя сфокусировать как самостоятельный интерактивный элемент;
- поле не получает собственного hover-состояния как интерактивный контрол.

Если пользователь нажимает на поле и не начинает drag, применяется только общее поведение карточки целиком.

### 4.7. Матрица прав и итоговое поведение
Поведение карточки определяется строго по этой матрице:

1. Если у пользователя есть право перемещать карточки и есть право открывать карточку:
   вся поверхность карточки является drag-зоной, а обычный клик без начала drag открывает карточку.
2. Если у пользователя есть право перемещать карточки, но нет права открывать карточку:
   вся поверхность карточки является drag-зоной, а обычный клик без начала drag не выполняет никакого дополнительного действия.
3. Если у пользователя нет права перемещать карточки, но есть право открывать карточку:
   карточка не является drag-зоной, а обычный клик открывает карточку.
4. Если у пользователя нет права перемещать карточки и нет права открывать карточку:
   карточка остаётся статическим неинтерактивным элементом.

Никаких других комбинаций поведения быть не должно.

### 4.8. Курсор и визуальные сигналы
Если пользователь может перетаскивать карточки, визуальный сигнал drag должен относиться ко всей поверхности карточки, а не к отдельной внутренней области.

Требования:
- курсор drag-поведения показывается для всей карточки;
- во время активного перетаскивания используется состояние “идёт перетаскивание” для всей карточки;
- внутренние элементы карточки не должны визуально выглядеть как самостоятельные интерактивные контролы;
- визуальные эффекты самой карточки во время drag сохраняют текущую логику перетаскивания карточки, если она уже есть.

### 4.9. Выделение текста и нативное перетаскивание содержимого
Чтобы не мешать drag-and-drop карточки:
- текст внутри карточки не должен выделяться в обычном сценарии захвата карточки мышью или пальцем;
- изображения и прочее внутреннее содержимое карточки не должны запускать нативный drag браузера;
- приоритет всегда остаётся за drag-and-drop самой карточки.

### 4.10. Клавиатурный сценарий
В рамках этой доработки не вводится отдельный новый клавиатурный сценарий перетаскивания карточки.

Фиксированное решение:
- эта доработка меняет pointer/touch-взаимодействие карточки;
- внутренние элементы карточки не участвуют в tab-порядке как самостоятельные элементы;
- существующее клавиатурное открытие карточки сохраняется только для самой карточки в тех случаях, когда карточку можно открыть;
- отдельного клавиатурного drag-сценария в этой спецификации не требуется.

## 5. Что не входит в доработку
В рамках этой спецификации не реализуются:
- клики по меткам;
- клики по адресу;
- клики по ссылочным пользовательским полям;
- клики по ответственному;
- клики по участникам;
- клики по счётчику комментариев;
- inline-редактирование карточки из списка;
- контекстные меню по внутренним элементам карточки;
- отдельные исключения для конкретных типов полей внутри карточки.

Если позже для какого-либо поля понадобится собственный клик-сценарий, это должно оформляться новой отдельной спецификацией, которая явно переопределит данный документ.

## 6. Приёмочные критерии

### 6.1. Старт drag из разных зон карточки
Карточка успешно начинает перетаскивание при старте из любой из следующих точек:
- из пустого места внутри карточки;
- из текста названия;
- из метки;
- из адреса;
- из ссылки;
- из даты;
- из select-поля;
- из текстового пользовательского поля;
- из блока ответственного;
- из блока участников;
- из счётчика комментариев.

Если элемент визуально находится внутри карточки, он должен подчиняться этому же правилу.

### 6.2. Отсутствие отдельной кликабельности внутренних элементов
При обычном клике по любому внутреннему элементу:
- не открывается отдельная ссылка;
- не открывается отдельный поповер;
- не запускается отдельное меню;
- не начинается отдельное редактирование;
- не выполняется никакое действие самого элемента.

### 6.3. Сохранение общего клика карточки
Если пользователь кликает по карточке и не доводит жест до начала drag:
- карточка открывается, если по правилам доступа она может быть открыта;
- карточка не открывается, если по правилам доступа она не может быть открыта.

### 6.4. Отсутствие ложного открытия после drag
Если пользователь начал drag из любой зоны карточки, затем отпустил карточку или отменил перетаскивание:
- карточка не открывается;
- не запускается действие никакого внутреннего элемента.

### 6.5. Отсутствие ручки `⋮⋮`
После внедрения в карточке не должно быть отдельной визуальной drag-ручки `⋮⋮`.

### 6.6. Единое поведение на мыши и таче
Все правила этой спецификации одинаково применяются:
- при взаимодействии мышью;
- при взаимодействии тачем.

## 7. Итоговое продуктовое правило
На доске карточка должна восприниматься как единый интерактивный блок:
- вся её поверхность используется для drag-and-drop;
- ни один внутренний элемент пока не имеет собственного клика;
- обычный клик без начала drag работает только как клик по самой карточке;
- ручка `⋮⋮` больше не нужна и не показывается.
