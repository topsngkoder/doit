# Спецификация: единая папка Яндекс.Диска на доску с доступом через приложение

## 1. Назначение документа
Документ фиксирует обязательные требования к хранению файлов доски в Яндекс.Диске через приложение.

Цель доработки:
- дать каждой доске одну общую папку в Яндекс.Диске;
- разрешить участникам доски загружать файлы в карточки и скачивать их только через интерфейс приложения;
- исключить прямую зависимость прав доступа от шаринга папки в интерфейсе Яндекс.Диска;
- сделать поведение подключения, загрузки, скачивания и удаления файлов однозначным.

В документе нет кода. Все решения ниже являются обязательными.

## 2. Приоритет документа
Для сценариев хранения файлов карточек в Яндекс.Диске этот документ имеет приоритет над любыми более ранними устными договорённостями и черновыми формулировками.

Если реализация противоречит этому документу, корректной считается реализация по этому документу.

## 3. Объект доработки
Под доработкой в этом документе понимаются:
- подключение Яндекс.Диска к конкретной доске;
- создание и использование общей папки доски в Яндекс.Диске;
- загрузка файлов в карточку через приложение;
- скачивание файлов карточки через приложение;
- удаление файлов карточки через приложение;
- хранение метаданных файлов в базе приложения;
- правила доступа и тексты ошибок;
- UI файловых полей `Яндекс диск` внутри карточки и UI настройки этого типа поля в `Поля доски`;
- сокращённый список готовых вложений на **отображении карточки на доске** (закрытая карточка в колонке) с возможностью скачивания по клику на **подпись имени файла**, а не на всю ширину строки (см. раздел 13.7).

Вне объекта доработки:
- прямой доступ участников доски к папке через интерфейс Яндекс.Диска;
- публичные ссылки Яндекс.Диска;
- предпросмотр файлов внутри приложения;
- редактирование файлов на стороне Яндекс.Диска;
- совместное редактирование документов;
- хранение файлов в Supabase Storage;
- drag-and-drop файлов на карточку без открытия карточки;
- версионирование файлов;
- синхронизация уже существующих файлов из чужих папок Яндекс.Диска.

## 4. Термины
В этом документе используются следующие термины:

- Интеграция доски: подключение одного аккаунта Яндекс.Диска к одной конкретной доске.
- Папка доски: корневая папка в Яндекс.Диске, выделенная одной доске.
- Вложение карточки: файл, прикреплённый к конкретной карточке и отображаемый в её интерфейсе.
- Поле доски типа `Яндекс диск`: пользовательское поле доски с `field_type = yandex_disk`, через которое в карточке отображаются и загружаются файлы.
- Файловое поле карточки: экземпляр поля типа `Яндекс диск` в интерфейсе конкретной карточки.
- Активная интеграция: состояние, в котором у доски есть валидная привязка к Яндекс.Диску и приложение может выполнять операции с файлами.
- Повторная авторизация: сценарий, в котором приложение должно заново получить действующий OAuth-доступ к уже подключённому Яндекс.Диску.

## 5. Зафиксированная архитектура

### 5.1. Провайдер хранения
Единственным внешним провайдером хранения файлов в рамках этой доработки является Яндекс.Диск.

Другие внешние провайдеры хранения в эту доработку не входят.

### 5.2. Модель привязки
Для одной доски поддерживается ровно одна интеграция с Яндекс.Диском.

Для одного board id допустима только одна активная запись подключения.

Подключение нескольких аккаунтов Яндекс.Диска к одной доске одновременно запрещено.

При этом на одной доске может существовать несколько пользовательских полей типа `Яндекс диск`.

Каждое вложение карточки обязано быть привязано к одному конкретному полю типа `Яндекс диск`.

### 5.3. Модель доступа
Все операции с файлами выполняются только через приложение.

Это означает:
- приложение само проверяет права пользователя на доску и карточку;
- приложение само обращается к API Яндекс.Диска;
- приложение само авторизует upload и выдаёт клиенту только одноразовый upload URL для заранее подготовленного пути файла;
- бинарное тело файла при загрузке в production не обязано проходить через HTTP body функции приложения: допустим и предпочтителен direct upload из браузера сразу в Яндекс.Диск по временному URL, полученному от приложения;
- пользователю не выдаются постоянные прямые доступы к папке доски;
- приложение не использует публичные ссылки Яндекс.Диска как основной механизм доступа.

### 5.4. Модель папок
У каждой доски есть одна корневая папка в Яндекс.Диске.

Обязательный путь корневой папки:
- `/doit/boards/<boardId>/`

Внутри корневой папки доски обязательно создаётся подпапка карточек:
- `/doit/boards/<boardId>/cards/`

Для каждой карточки, в которую был загружен хотя бы один файл, создаётся отдельная подпапка:
- `/doit/boards/<boardId>/cards/<cardId>/`

### 5.5. Модель хранения файлов карточки
Каждый файл карточки хранится в подпапке своей карточки.

Обязательный путь файла:
- `/doit/boards/<boardId>/cards/<cardId>/<attachmentId><extension>`

Правила имени файла в Яндекс.Диске:
- в имени файла на стороне Яндекс.Диска используется только идентификатор вложения и расширение;
- исходное имя файла в путь Яндекс.Диска не включается;
- исходное имя файла хранится только в базе приложения;
- если расширение определить невозможно, файл сохраняется без расширения.

## 6. Зафиксированные продуктовые решения

### 6.1. Что считается вложением
Вложением считается файл, загруженный в конкретную карточку через конкретное поле типа `Яндекс диск` и связанный с этой карточкой и этим полем в базе приложения.

Файл доски без привязки к карточке в этой доработке не поддерживается.

### 6.2. Где доступна загрузка
Загрузка файлов поддерживается только внутри открытой карточки.

Поддерживаются следующие точки входа:
- кнопка `Добавить файлы` внутри конкретного поля типа `Яндекс диск`;
- drag-and-drop файлов в зону загрузки внутри конкретного поля типа `Яндекс диск`.

Перетаскивание файла на закрытую карточку в колонке, на саму колонку или на фон доски не поддерживается.

### 6.3. Где доступно скачивание
Скачивание файлов доступно:
- из списка вложений внутри конкретного поля типа `Яндекс диск` в **открытой** карточке;
- из сокращённого списка на **отображении карточки на доске** по правилам раздела 13.7.

В обоих случаях используется тот же серверный сценарий скачивания: приложение получает у Яндекса одноразовый временный URL, отдаёт пользователю файл с именем из базы (`original_file_name`, см. раздел 11.5); постоянные прямые URL Яндекс.Диска пользователю не показываются.

Отдельного глобального файлового менеджера доски в этой доработке нет.

### 6.4. Где доступно удаление
Удаление файла доступно только из списка вложений внутри конкретного поля типа `Яндекс диск` в карточке.

Переименование вложений не поддерживается.

## 7. Таблицы и обязательные данные

### 7.1. Привязка Яндекс.Диска к доске
В системе должна существовать отдельная сущность привязки Яндекс.Диска к доске.

Обязательные поля привязки:
- идентификатор записи;
- идентификатор доски;
- идентификатор аккаунта Яндекса;
- логин аккаунта Яндекса;
- путь корневой папки доски;
- зашифрованный access token;
- зашифрованный refresh token;
- время истечения access token;
- статус интеграции;
- пользователь, который выполнил подключение;
- дата и время создания;
- дата и время последнего обновления;
- дата и время последней успешной авторизации;
- текст последней ошибки интеграции, если она есть.

Допустимые статусы интеграции:
- `active`
- `reauthorization_required`
- `disconnected`
- `error`

Для одной доски одновременно допустима только одна запись со статусом `active`.

### 7.2. Вложения карточек
В системе должна существовать отдельная сущность вложений карточек.

Обязательные поля вложения:
- идентификатор вложения;
- идентификатор доски;
- идентификатор карточки;
- идентификатор определения поля доски типа `Яндекс диск`;
- провайдер хранения;
- путь файла в Яндекс.Диске;
- исходное имя файла;
- MIME-тип;
- размер файла в байтах;
- пользователь, загрузивший файл;
- дата и время загрузки;
- статус вложения.

Провайдер хранения для этой доработки всегда равен Яндекс.Диску.

Каждое вложение обязано ссылаться на существующее определение поля доски с типом `yandex_disk`.

Допустимые статусы вложения:
- `uploading`
- `ready`
- `failed`

В интерфейсе карточки отображаются только вложения со статусом `ready`.

## 8. Права доступа

### 8.1. Подключение интеграции
Подключать, переподключать и отключать Яндекс.Диск для доски может только владелец доски.

Ни администратор доски, ни обычный участник, ни наблюдатель не могут выполнять эти действия, если они не являются владельцем доски.

### 8.2. Просмотр списка файлов
Пользователь видит файловые поля карточки и список готовых вложений только тогда, когда он имеет право открыть эту карточку и просматривать её содержимое.

### 8.3. Загрузка файлов
Загружать файлы в карточку может только пользователь, который имеет право редактировать содержимое этой карточки по тем же правилам, что и редактирование описания и пользовательских полей карточки.

### 8.4. Скачивание файлов
Скачивать файлы карточки может любой пользователь, который имеет право просматривать содержимое этой карточки.

### 8.5. Удаление файлов
Удалять файлы карточки может только пользователь, который имеет право редактировать содержимое этой карточки по тем же правилам, что и редактирование описания и пользовательских полей карточки.

### 8.6. Независимость от прав Яндекс.Диска
Права доступа к файлам определяются только приложением.

Факт того, что пользователь сам не имеет доступа к папке в интерфейсе Яндекс.Диска, не влияет на его право скачать или загрузить файл через приложение.

## 9. Подключение Яндекс.Диска к доске

### 9.1. Точка входа
Подключение интеграции и настройка файловых точек входа выполняются из настроек конкретной доски, в разделе `Поля доски`.

В выпадающем списке `Тип` при создании или редактировании поля доски должен существовать вариант `Яндекс диск`.

Создание поля этого типа добавляет на карточки отдельное файловое поле, связанное с интеграцией Яндекс.Диска уровня доски.

На одной доске допускается создание нескольких полей типа `Яндекс диск`.

Отдельная обязательная кнопка или отдельный обязательный раздел `Яндекс.Диск` вне `Поля доски` в этой доработке не требуются.

### 9.2. OAuth-модель
Интеграция использует одно приложение OAuth Яндекса, зарегистрированное для приложения Doit.

При подключении владелец доски проходит OAuth-авторизацию Яндекса и подтверждает доступ приложения к своему Яндекс.Диску.

### 9.3. Создание структуры папок
После первого успешного подключения приложение обязано:
- проверить существование папки `/doit/`;
- проверить существование папки `/doit/boards/`;
- создать папку `/doit/boards/<boardId>/`, если её нет;
- создать папку `/doit/boards/<boardId>/cards/`, если её нет.

Если указанные папки уже существуют, они повторно не создаются и не переименовываются.

### 9.4. Повторное подключение
Если у доски уже есть активная интеграция, повторное подключение через UI не создаёт вторую интеграцию.

Повторное подключение в этом случае означает только обновление токенов существующей интеграции.

### 9.5. Смена аккаунта Яндекс.Диска
Смена подключённого аккаунта Яндекс.Диска для доски запрещена, если на доске существует хотя бы одно готовое вложение карточки.

В этом случае пользователь получает ошибку:

`Нельзя сменить Яндекс.Диск для доски, пока в карточках есть файлы.`

### 9.6. Отключение интеграции
Отключение интеграции переводит запись подключения в статус `disconnected`.

При отключении:
- файлы из Яндекс.Диска автоматически не удаляются;
- записи вложений карточек автоматически не удаляются;
- новые загрузки становятся недоступны;
- скачивание существующих файлов становится недоступно до повторной авторизации этой же интеграции.

### 9.7. Повторная авторизация
Если access token недействителен, приложение обязано попытаться обновить его через refresh token.

Если refresh token также недействителен или отозван, интеграция переводится в статус `reauthorization_required`.

В этом состоянии:
- загрузка файлов запрещена;
- скачивание файлов запрещено;
- владелец доски видит предложение повторно авторизовать интеграцию;
- остальные пользователи видят только сообщение о недоступности хранилища.

## 10. Правила загрузки файлов

### 10.1. Поддерживаемый сценарий
Пользователь может загрузить один или несколько файлов в открытую карточку через кнопку выбора файлов или drag-and-drop в зону загрузки конкретного поля типа `Яндекс диск`.

### 10.2. Ограничения загрузки
Обязательные ограничения:
- максимум 20 файлов за одну операцию;
- максимум 200 готовых вложений на одну карточку;
- максимальный размер одного файла: **1 ГБ** (в приложении проверка по **1 ГиБ**, 1024³ байт);
- пустой файл загружать нельзя.

### 10.3. Типы файлов
Допускаются любые типы файлов.

Предпросмотр по MIME-типу не реализуется.

Все скачивания выполняются как скачивание файла, а не как встроенное открытие в приложении.

### 10.4. Последовательность загрузки
Для каждого файла сценарий обязателен в таком порядке:

1. Приложение проверяет права пользователя на загрузку в карточку.
2. Приложение проверяет активность интеграции доски.
3. Приложение определяет технический путь файла в Яндекс.Диске и при необходимости создаёт подпапку карточки.
4. Приложение создаёт запись вложения со статусом `uploading`.
5. Приложение получает у Яндекс.Диска одноразовый URL для загрузки в заранее зафиксированный путь файла.
6. Приложение возвращает этот URL клиенту как часть краткоживущего upload-intent для конкретного файла.
7. Браузер загружает байты файла напрямую в Яндекс.Диск по этому URL, без проксирования полного тела файла через обязательный route handler приложения.
8. После подтверждённого успеха direct upload клиент вызывает короткий server-side сценарий завершения, а приложение переводит запись вложения в статус `ready`.

Если любой шаг после пункта 4 не завершился успехом, запись вложения переводится в статус `failed`.

Примечание по deployment-модели:
- продуктовое правило «загрузка через приложение» в этой доработке означает server-side авторизацию, валидацию, подготовку пути, выдачу краткоживущего upload URL и финализацию статуса;
- для production-сред с жёстким лимитом размера request body у функций приложения direct upload browser -> Yandex Disk считается целевым сценарием, а не отклонением от требований.

### 10.5. Создание подпапки карточки
Если для карточки ещё не существует подпапка в Яндекс.Диске, приложение обязано создать её перед первой загрузкой файла в эту карточку.

### 10.6. Поведение после успеха
После успешной загрузки:
- карточка остаётся открытой;
- список вложений обновляется без смены маршрута;
- новое вложение появляется в списке файлов соответствующего поля карточки;
- пользователь видит исходное имя файла, размер, автора загрузки и дату загрузки.

### 10.7. Поведение при частичном успехе
Если пользователь выбрал несколько файлов и часть загрузилась успешно, а часть завершилась ошибкой:
- успешно загруженные файлы остаются в карточке;
- неуспешные файлы не отображаются как готовые вложения;
- пользователь получает отдельное сообщение об ошибке для каждого неуспешного файла.

## 11. Правила скачивания файлов

### 11.1. Общий принцип
Скачивание выполняется только через приложение после проверки права на просмотр карточки.

### 11.2. Временный доступ
Для скачивания приложение обязано получать у Яндекс.Диска временный URL на конкретный файл.

Постоянный прямой URL файла в Яндекс.Диске пользователю не сохраняется и не показывается.

### 11.3. Время жизни ссылки скачивания
Временный URL для скачивания считается одноразовым прикладным доступом и должен использоваться сразу после выдачи.

Приложение не кэширует такой URL дольше 5 минут.

### 11.4. Отсутствующий файл у провайдера
Если запись вложения есть в базе, но файла больше нет в Яндекс.Диске, операция скачивания завершается ошибкой:

`Файл не найден в Яндекс.Диске.`

Такое вложение остаётся в базе до явного удаления пользователем с правом редактирования карточки.

### 11.5. Имя файла при скачивании
Имя файла, под которым браузер или ОС сохраняет скачанный файл пользователю, **должно совпадать с исходным именем файла**, хранящимся в базе приложения (`original_file_name`), то есть с тем же именем, что показывается в списке вложений в карточке и на отображении карточки на доске.

Техническое имя файла на стороне Яндекс.Диска (`<attachmentId><extension>`) пользователю при скачивании **не** подставляется.

Для этого приложение может проксировать байты файла от провайдера к клиенту и выставлять заголовок `Content-Disposition` (включая поддержку не-ASCII имён, например параметр `filename*` UTF-8 по RFC 5987). Временный URL Яндекса по-прежнему используется только на сервере и не считается постоянной ссылкой для пользователя (согласовано с разделами 11.2 и 11.3).

## 12. Правила удаления файлов

### 12.1. Общий принцип
Удаление файла выполняется только из интерфейса карточки после проверки права на редактирование содержимого карточки.

### 12.2. Порядок удаления
При удалении приложение обязано:
- проверить право пользователя;
- удалить файл из Яндекс.Диска;
- удалить запись вложения из базы приложения.

### 12.3. Поведение при отсутствии файла у провайдера
Если в момент удаления файл уже отсутствует в Яндекс.Диске, операция удаления считается успешной очисткой.

В этом случае запись вложения всё равно удаляется из базы приложения.

### 12.4. Удаление карточки
При удалении карточки все её вложения должны удаляться и из базы приложения, и из Яндекс.Диска.

Если часть файлов в Яндекс.Диске уже отсутствует, удаление карточки всё равно должно завершаться успешно при условии удаления соответствующих записей вложений из базы.

## 13. Поведение UI карточки

### 13.1. Файловые поля в карточке
В карточке на вкладке деталей поля типа `Яндекс диск` отображаются внутри блока пользовательских полей карточки.

Если на доске создано несколько полей типа `Яндекс диск`, в карточке отображается отдельный файловый интерфейс для каждого такого поля.

### 13.2. Состав файлового поля
Каждое поле типа `Яндекс диск` в карточке содержит:
- название поля;
- кнопку `Добавить файлы`, если у пользователя есть право загрузки;
- drag-and-drop зону загрузки, если у пользователя есть право загрузки;
- пустое состояние, если для этого поля файлов нет;
- список готовых вложений этого поля, если файлы есть;
- во время активной загрузки — индикатор с прогрессом и скоростью по правилам раздела 13.5.

### 13.3. Пустое состояние
Если у конкретного поля типа `Яндекс диск` нет готовых вложений:
- показывается пустое состояние;
- если у пользователя есть право загрузки, пустое состояние содержит явный призыв перетащить файлы или выбрать их кнопкой;
- если у пользователя нет права загрузки, пустое состояние не содержит интерактивных элементов.

### 13.4. Элемент списка вложений
Каждое готовое вложение в списке обязано показывать:
- исходное имя файла;
- размер файла в человекочитаемом виде;
- имя пользователя, который загрузил файл;
- дату и время загрузки;
- кнопку `Скачать`, если у пользователя есть право просмотра (скачивание с тем же исходным именем файла, что в списке — раздел 11.5);
- кнопку `Удалить`, если у пользователя есть право редактирования содержимого карточки.

### 13.5. Состояние загрузки
Во время активной загрузки файлов пользователь видит локальный индикатор загрузки.

Вложения со статусом `uploading` не отображаются в постоянном списке готовых файлов конкретного поля.

Дополнительно, пока для конкретного поля типа `Яндекс диск` идёт подготовка upload-intent приложением и дальнейшая прямая загрузка файла в Яндекс.Диск, интерфейс этого поля обязан показывать:

- **прогресс** по каждому файлу в текущей операции: доля выполнения в процентах и/или передано из известного объёма (`передано / всего` в человекочитаемых единицах); для direct upload целевой источник этого прогресса — фактическая передача байтов из браузера в Яндекс.Диск; если на отдельном коротком server-side этапе (подготовка upload-intent или финализация) объём не измеряется, для этого короткого этапа допускается неопределённый индикатор (indeterminate), но не вместо байтового прогресса основной передачи;
- **скорость** передачи: текущая скорость загрузки в человекочитаемом виде (например, `МиБ/с` или `КиБ/с`), сглаженная по короткому окну, чтобы значение не «дёргалось» на каждом тике; целевой источник — фактическая передача из браузера в Яндекс.Диск; при отсутствии данных для расчёта скорость можно временно скрывать или показывать прочерк, но не подставлять вымышленные числа.

При пакетной загрузке нескольких файлов:

- пользователь видит, какой файл сейчас загружается (имя из исходного выбора);
- по завершении или ошибке одного файла прогресс и скорость для него исчезают, для следующего файла показываются заново.

Требования к прогрессу и скорости относятся к **открытой карточке** и **конкретному полю**, в котором запущена загрузка; они не меняют правило, что в постоянном списке остаются только вложения со статусом `ready`.

### 13.6. Недоступная интеграция
Если у доски нет активной интеграции Яндекс.Диска:
- каждое поле типа `Яндекс диск` остаётся видимым;
- интерактивные действия загрузки и скачивания в этих полях недоступны;
- в каждом таком поле показывается причина недоступности.

### 13.7. Вложения на отображении карточки на доске
Под **отображением карточки** понимается карточка в колонке доски в закрытом виде (не модалка с деталями).

На отображении карточки для всех полей типа `Яндекс диск` показывается **единый** список готовых вложений карточки (только `ready`), если у пользователя есть право просмотра содержимого этой карточки:

- **Не более трёх строк** файлов. Если готовых вложений больше трёх, на отображении карточки показываются только первые три; остальные доступны только в **открытой** карточке в полных списках по полям.
- **Каждое вложение — не более одной строки** текста (исходное имя файла; длинные имена обрезаются визуально, например многоточием, без переноса на вторую строку).
- **Скачивание** соответствующего файла через приложение запускается по клику **только на отображаемую подпись имени файла** в строке (продуктово — дополнительная «кнопка» на карточке), а **не** на пустую часть строки по ширине карточки: эта зона относится к самой карточке (открытие, перетаскивание). Проверки те же, что у кнопки `Скачать` в открытой карточке — право просмотра карточки и активная интеграция (раздел 13.4). При недоступной интеграции подписи на отображении карточки **не** инициируют скачивание (элементы неактивны или список не показывается — на усмотрение реализации при условии отсутствия обхода правил 13.6).
- Загрузка и удаление файлов с отображения карточки на доске **не** выполняются; они по-прежнему только внутри открытой карточки (раздел 6.2).

**Порядок** вложений в сокращённом списке на доске: в порядке определений полей типа `Яндекс диск` на доске (как в UI открытой карточки), внутри каждого поля — в том же порядке, что и полный список этого поля в открытой карточке.

## 14. Поведение UI настроек доски

### 14.1. Настройка поля
В модалке `Поля доски` пользователь должен иметь возможность выбрать тип поля `Яндекс диск`.

На одной доске можно создать несколько полей такого типа.

### 14.2. Что видно владельцу доски
Владелец доски в контексте полей типа `Яндекс диск` видит состояние интеграции и управляющие действия.

Состояние интеграции Яндекс.Диска при этом обязано отображаться как одно из:
- не подключено;
- подключено;
- требуется повторная авторизация;
- отключено;
- ошибка.

В подключённом состоянии владелец доски видит:
- логин подключённого Яндекс-аккаунта;
- путь корневой папки доски;
- дату последней успешной авторизации;
- действие повторной авторизации;
- действие отключения.

### 14.3. Что видно не владельцу доски
Пользователь, который не является владельцем доски:
- при работе с полями типа `Яндекс диск` видит только факт наличия или отсутствия доступной интеграции;
- не видит токены, служебные ошибки OAuth и owner-only действия подключения;
- не может изменить интеграцию.

## 15. Тексты ошибок
Ниже зафиксированы обязательные пользовательские сообщения.

### 15.1. Ошибки прав и авторизации
- `Нужна авторизация.`
- `У вас нет права подключать Яндекс.Диск для этой доски.`
- `У вас нет права загружать файлы в эту карточку.`
- `У вас нет права скачивать файлы этой карточки.`
- `У вас нет права удалять файлы этой карточки.`

### 15.2. Ошибки интеграции
- `Для этой доски не подключён Яндекс.Диск.`

- `Не удалось создать папку доски в Яндекс.Диске.`
- `Нельзя сменить Яндекс.Диск для доски, пока в карточках есть файлы.`

### 15.3. Ошибки файлов
- `Файл пустой.`
- `Файл слишком большой. Максимальный размер файла — 1 ГБ.`

- `Файл не найден в Яндекс.Диске.`
- `Не удалось загрузить файл в Яндекс.Диск.`
- `Не удалось скачать файл из Яндекс.Диска.`
- `Не удалось удалить файл из Яндекс.Диска.`

## 16. Очистка и консистентность

### 16.1. Записи со статусом failed
Вложения со статусом `failed` не показываются пользователю в списке файлов карточки.

Такие записи должны автоматически удаляться служебной очисткой не позднее чем через 24 часа после возникновения ошибки.

### 16.2. Осиротевшие файлы
Если в Яндекс.Диске существует файл в папке доски, но для него нет записи вложения в базе приложения, такой файл считается осиротевшим.

Осиротевшие файлы должны автоматически удаляться служебной очисткой не позднее чем через 24 часа после обнаружения.

### 16.3. Осиротевшие подпапки карточек
Пустые подпапки карточек допускаются и автоматически не удаляются.

## 17. Что не должно меняться
Эта доработка не должна менять:
- текущую модель прав на просмотр и редактирование карточки;
- существующую логику описания карточки;
- существующую логику пользовательских полей карточки;
- существующую логику комментариев карточки;
- drag-and-drop карточек и колонок на доске;
- существующую маршрутизацию доски;
- существующее хранение фоновых изображений доски;
- существующее хранение аватаров пользователей.

## 18. Недопустимое поведение
Считаются ошибкой следующие состояния:
- одна доска одновременно использует несколько активных аккаунтов Яндекс.Диска;
- участники доски обязаны получать доступ к папке через интерфейс Яндекс.Диска вместо приложения;
- файл карточки сохраняется вне папки своей доски;
- файл карточки сохраняется вне подпапки своей карточки;
- исходное имя файла используется как основное имя файла в Яндекс.Диске;
- пользователь без права просмотра карточки может скачать файл;
- пользователь без права редактирования содержимого карточки может загрузить файл;
- пользователь без права редактирования содержимого карточки может удалить файл;
- отсутствие интеграции приводит к скрытию файлового поля вместо явного сообщения;
- скачивание файла требует публичной ссылки Яндекс.Диска;
- отключение интеграции автоматически удаляет файлы пользователя из Яндекс.Диска;
- смена подключённого Яндекс-аккаунта допускается при наличии готовых вложений карточек;
- drag-and-drop на доску вне открытой карточки начинает загрузку файлов;
- failed-вложения отображаются как нормальные готовые файлы;
- файлы из разных полей типа `Яндекс диск` смешиваются в один общий список без привязки к полю;
- прямой URL Яндекс.Диска сохраняется в интерфейсе как постоянная ссылка на файл;
- production-upload больших файлов требует проксирования полного тела файла через request body serverless-функции приложения, из-за чего нарушается поддерживаемый лимит размера;
- скачивание сохраняет файл под техническим именем на Яндекс.Диске вместо исходного имени из базы.

## 19. Критерии приёмки
Задача считается выполненной, если одновременно соблюдаются все условия:

1. Каждая доска может иметь не более одной активной интеграции Яндекс.Диска.
2. Для подключённой доски существует корневая папка строго по пути `/doit/boards/<boardId>/`.
3. Файлы карточек сохраняются строго в подпапках `/doit/boards/<boardId>/cards/<cardId>/`.
4. Участники доски загружают и скачивают файлы только через приложение; для upload допустим direct upload browser -> Yandex Disk, если именно приложение выполнило авторизацию, валидацию, выдачу краткоживущего upload URL и финализацию статуса вложения.
5. В `Поля доски` доступен тип поля `Яндекс диск`, и владелец доски может подключить, переподключить и отключить интеграцию из этого сценария.
6. Пользователь с правом редактирования содержимого карточки может загрузить файл в открытую карточку.
7. Пользователь с правом просмотра карточки может скачать файл из открытой карточки.
8. Пользователь без права редактирования содержимого карточки не может загрузить или удалить файл.
9. При отсутствии активной интеграции поле типа `Яндекс диск` в карточке остаётся видимым и показывает понятную причину недоступности.
10. При удалении карточки её вложения удаляются из базы приложения и из Яндекс.Диска.
11. При отключении интеграции файлы физически не удаляются из Яндекс.Диска автоматически.
12. При наличии хотя бы одного готового вложения доски смена подключённого Яндекс-аккаунта запрещена.
13. На одной доске можно создать несколько полей типа `Яндекс диск`, и файлы каждого такого поля в карточке отображаются отдельно.
14. Во время загрузки файлов в поле типа `Яндекс диск` пользователь видит прогресс (процент и/или байты из известного объёма) и сглаженную скорость передачи; при выборе нескольких файлов видно текущий файл и состояние по очереди.
15. На отображении карточки на доске пользователь с правом просмотра видит не более трёх однострочных имён готовых вложений (порядок по разделу 13.7); по клику **по подписи имени файла** начинается скачивание через приложение при активной интеграции; клик по пустой части строки ведёт себя как клик по карточке (открытие и перетаскивание, а не скачивание); при более чем трёх вложениях остальные доступны только в открытой карточке.
16. При скачивании готового вложения сохраняемое имя файла совпадает с исходным именем из базы (`original_file_name`), а не с именем объекта на Яндекс.Диске (раздел 11.5).
