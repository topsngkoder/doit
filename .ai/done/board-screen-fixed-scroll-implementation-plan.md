## Назначение
Этот документ предназначен для AI-агента, который будет внедрять требования из `.ai/board-screen-fixed-scroll-specification.md`.

Цель: перестроить экран `/boards/{boardId}` так, чтобы:
- фон доски был визуально закреплён относительно viewport;
- верхняя полоса приложения не скроллилась;
- полоса доски с названием не скроллилась;
- панель действий над колонками не скроллилась;
- существовал ровно один основной scroll-container;
- и вертикальная, и горизонтальная прокрутка жили только внутри viewport колонок.

## Обязательные инварианты реализации
- Нельзя оставлять scroll у `body`.
- Нельзя делать верхние панели `sticky` внутри внешнего прокручиваемого контейнера, если из-за этого продолжает скроллиться страница целиком.
- Нельзя оставлять горизонтальный scroll на уровне внутренних рядов колонок, если родительский viewport колонок не является тем самым единственным scroll-container.
- Нельзя закреплять фон доски относительно прокручиваемого контейнера колонок.
- Нельзя переводить колонки в вертикальный стек на узких ширинах.
- Нельзя вводить отдельный вертикальный scroll внутри каждой колонки.
- Нельзя менять бизнес-логику доски, набор данных, права доступа, тексты и порядок колонок.
- Если для соблюдения спецификации потребуется неочевидное UX-решение, которого нет в спецификации, агент обязан остановиться и спросить пользователя.

## Основное архитектурное решение

### 1. Каркас экрана
Экран доски должен быть собран как shell на всю высоту viewport браузера.

Предпочтительная модель:
- корневой board-shell с высотой `100dvh` или эквивалентной безопасной viewport-высотой;
- 4 вертикальные зоны сверху вниз:
  1. глобальный header;
  2. полоса доски;
  3. панель действий над колонками;
  4. viewport колонок.

### 2. Как считать высоту viewport колонок
Предпочтение: CSS-layout без ручной математики через `grid`/`flex` + `auto auto auto minmax(0, 1fr)`.

То есть:
- высоты верхних трёх зон остаются естественными;
- viewport колонок получает остаток высоты;
- при переносах строк и динамическом изменении высоты верхних зон доступная высота viewport колонок пересчитывается автоматически.

JS-измерения или `ResizeObserver` разрешены только как fallback, если чистый CSS не даёт гарантированно убрать внешний scroll.

### 3. Единственный scroll-container
Единственным основным scroll-container должен стать контейнер viewport колонок.

Он обязан одновременно владеть:
- вертикальной прокруткой;
- горизонтальной прокруткой.

Внутренний DOM должен быть таким, чтобы:
- сам viewport имел `overflow: auto`;
- внутри него лежала горизонтальная лента колонок;
- колонкам и родительским shell-узлам не приходилось создавать собственный конкурирующий scroll для страницы.

### 4. Фон доски
Фон доски должен быть вынесен в слой, закреплённый относительно viewport board-screen, а не относительно движущегося содержимого колонок.

Обязательные свойства фонового изображения:
- `background-size: cover`;
- `background-position: center`;
- `background-repeat: no-repeat`;
- отсутствие параллакса;
- отсутствие смещения при любом scroll внутри viewport колонок.

### 5. Предпочтительный путь рефакторинга
Чтобы не смешивать фиксированные зоны и scroll-зону, лучше разделить текущий `BoardCanvas` на более явные роли:
- shell/каркас экрана;
- фиксированная панель действий над колонками;
- viewport колонок;
- компонент ленты колонок (`BoardColumnsDnD`) без владения page-level scroll.

Решение по структуре файлов на T02:
- Новый shell-файл **не создаётся** на этом шаге (избежание лишней миграции структуры).
- Точка входа shell остаётся `BoardCanvas`; разрезание ответственности выполняется последовательно в `T03`–`T08`.

### Фаза 3. Высотный каркас board-страницы
- Привести `/boards/[boardId]` к полноценной высоте viewport.
- Убедиться, что board-screen может занимать остаток высоты под глобальным header без внешнего вертикального scroll.
- Убрать классы и обёртки, которые мешают `min-height: 0` и нормальному сжатию flex/grid-детей.
- Если нужно, локально скорректировать обвязку внутри `layout.tsx`, чтобы board route не зависел от `max-w-5xl` как от ограничителя высоты.
- После этого шага корневой экран доски должен уметь жить в модели `верхние auto + viewport 1fr`.

### Фаза 4. Выделение полосы доски как фиксированной зоны
- Оставить заголовок доски, участников и действия доски в отдельной неперемещаемой зоне.
- Убедиться, что этот блок не находится внутри scroll-container колонок.
- Исключить ситуации, когда горизонтальный scroll колонок двигает эту зону влево/вправо.
- Исключить ситуации, когда вертикальный scroll колонок уводит этот блок вверх.

### Фаза 5. Выделение панели действий над колонками
- Перестать рендерить панель действий как часть общего прокручиваемого содержимого.
- Вынести `AddBoardColumnButton` и связанные служебные сообщения в отдельную фиксированную зону над viewport колонок.
- Сохранить правило спецификации:
  - если в панели нет видимых элементов, она не должна занимать высоту;
  - если есть ошибка/статус/кнопка, панель участвует в высотной формуле.
- Если служебное сообщение живёт внутри `AddBoardColumnButton`, проверить, достаточно ли текущей структуры; при необходимости вынести контейнер сообщений на уровень панели.

### Фаза 6. Перенос фона доски в фиксированный слой
- Перестроить `BoardBackgroundFrame`, чтобы фон не был привязан к движению содержимого колонок.
- Фон должен покрывать весь board-screen, включая фиксированные верхние зоны доски и viewport колонок.
- Слой фона должен быть визуально под контентом и не перехватывать pointer events.
- Для режима `backgroundType === "image"` сохранить текущую логику signed URL и кеша.
- Для режима без изображения сохранить фиксированный базовый фон.

### Фаза 7. Создание единственного viewport колонок
- Ввести единый контейнер viewport колонок на уровне shell.
- Этот контейнер должен:
  - занимать остаток высоты;
  - иметь `min-height: 0`;
  - быть единственным page-level scroll-container;
  - владеть `overflow: auto`.
- Все scroll-сценарии страницы доски должны проходить через него.
- Родители viewport не должны иметь собственного `overflow`, который конкурирует со scroll viewport.

### Фаза 8. Горизонтальная прокрутка только внутри viewport колонок
- Убрать top-level `overflow-x-auto` из веток рендера `BoardColumnsDnD`, если именно они сейчас являются конкурирующим scroll-контейнером.
- Оставить `BoardColumnsDnD` ответственным за ленту колонок, а не за page-level scroll.
- Лента колонок внутри viewport должна иметь ширину по содержимому:
  - колонки идут в один ряд;
  - не переносятся;
  - на узких экранах появляется горизонтальная прокрутка именно у viewport.
- Проверить все ветки рендера в `BoardColumnsDnD`:
  - статический режим;
  - режим сортировки колонок;
  - прочие ветки, где повторяется один и тот же контейнер.

### Фаза 9. Вертикальная прокрутка только внутри viewport колонок
- Убедиться, что при большом числе карточек движется только содержимое viewport колонок.
- Сами колонки не должны получать отдельный page-level vertical scroll.
- Не добавлять `overflow-y-auto` на каждую колонку.
- Проверить, что высота карточек/колонок естественно растягивает горизонтальную ленту, а не ломает scroll-модель.

### Фаза 10. Динамическая высота при изменении верхних зон
- Проверить сценарии:
  - уменьшение ширины окна;
  - перенос элементов в header/board bar;
  - появление ошибки/служебного сообщения в панели действий;
  - изменение zoom.
- Убедиться, что при этих изменениях viewport колонок автоматически пересчитывает доступную высоту.
- Если после CSS-рефакторинга где-то всё ещё появляется scroll у `body`, локализовать причину.
- Только если CSS-модель не покрывает кейс, добавить измерение высот через `ResizeObserver`; иначе не усложнять реализацию.

### Фаза 11. Проверка фонового поведения на обеих осях
- Проверить, что при вертикальном scroll фон не двигается.
- Проверить, что при горизонтальном scroll фон не двигается.
- Проверить, что прокрутка колонок не меняет точку привязки изображения.
- Проверить, что нет параллакса.

### Фаза 12. Защита от побочных scroll-контейнеров
- Проверить board-shell на предмет `overflow-hidden`, `overflow-auto`, `overflow-x-auto`, `overflow-y-auto`.
- Разделить допустимые локальные scroll-контейнеры и недопустимые page-level scroll-контейнеры.
- Допустимы:
  - modal body;
  - dropdown/popover списки;
  - локальные списки внутри popup.
- Недопустимы:
  - scroll у `body`;
  - scroll у root board-page;
  - scroll у `BoardBackgroundFrame`;
  - отдельный scroll каждой колонки;
  - отдельный конкурентный scroll у оболочки `BoardColumnsDnD`.

### Фаза 13. Ручная проверка адаптивности
- Проверить широкий экран: верхние зоны фиксированы, горизонтальный scroll отсутствует или работает только внутри viewport.
- Проверить узкий экран: колонки остаются горизонтальной лентой, не складываются в столбец.
- Проверить маленькую высоту окна: появляется только вертикальный scroll viewport колонок.
- Проверить комбинацию малого width и малого height: обе оси scroll работают только внутри viewport.

### Фаза 14. Линтеры и финальная стабилизация
- Проверить изменённые файлы через `ReadLints`.
- Исправить новые диагностические ошибки, если они появились из-за рефакторинга.
- Повторно пройтись по `overflow`, `min-h-0`, `h-full`, `100vh`, `100dvh`, `sticky`, `fixed`.
- Убедиться, что после финальной версии нет конфликтующих правил, оставшихся от промежуточной структуры.

## Чек-лист приёмки по спецификации
- На `/boards/{boardId}` нет scroll у `body`.
- Верхняя полоса приложения всегда остаётся видимой и неподвижной.
- Полоса доски всегда остаётся видимой и неподвижной.
- Панель действий над колонками не скроллится вместе с колонками.
- Фон доски не двигается ни по вертикали, ни по горизонтали.
- Единственный основной scroll-container страницы доски — viewport колонок.
- При нехватке ширины горизонтальная прокрутка появляется только внутри viewport колонок.
- При нехватке высоты вертикальная прокрутка появляется только внутри viewport колонок.
- При нехватке ширины и высоты обе прокрутки живут только внутри viewport колонок.
- Верхние панели не смещаются влево/вправо при горизонтальной прокрутке колонок.
- На узких экранах колонки не перестраиваются в один столбец.
- Внутри отдельной колонки не появляется собственный вертикальный scroll-container.
