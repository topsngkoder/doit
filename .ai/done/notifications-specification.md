# Спецификация изменений уведомлений: email + браузер

## 1. Назначение документа
Этот документ фиксирует конечные требования к уведомлениям и **переопределяет** все противоречащие пункты в ранее созданных спецификациях проекта.

Цель изменения:
- оставить только два пользовательских канала уведомлений: **браузер** и **email**;
- убрать из экрана настроек уведомлений временную зону;
- убрать из экрана настроек уведомлений Telegram;
- добавить отдельные галочки для email-уведомлений по каждому типу события;
- добавить два новых типа уведомлений:
  - **“Ваша карточка готова”**
  - **“Ваша карточка в работе”**
- добавить кнопку **“Включить уведомления в браузере”**, которая при нажатии запрашивает у браузера разрешение на показ системных уведомлений.

## 2. Фиксированные решения

### 2.1. Каналы уведомлений
В системе существуют только два пользовательских канала:
- `browser`
- `email`

Других пользовательских каналов в MVP нет.

### 2.2. Telegram
Telegram-уведомления:
- не отображаются в UI;
- не настраиваются пользователем;
- не участвуют в логике доставки;
- не учитываются в матрице типов уведомлений;
- не упоминаются на экране настроек уведомлений.

### 2.3. Временная зона
Поле выбора временной зоны полностью убирается с экрана настроек уведомлений.

В рамках этой спецификации:
- пользователь **не выбирает** временную зону для уведомлений;
- уведомления **не зависят** от временной зоны пользователя;
- логика тихих часов **не используется**;
- любые ранее существовавшие настройки временной зоны и тихих часов считаются удалёнными из пользовательского сценария и не влияют на поведение системы.

### 2.4. Что считать “браузерными уведомлениями”
Канал `browser` состоит из двух частей:
1. запись во внутренний центр уведомлений приложения;
2. нативное системное уведомление браузера через Browser Notification API, если браузер выдал разрешение.

Следствие:
- если для события канал `browser` включен, запись во внутреннем центре уведомлений создаётся всегда;
- нативное всплывающее браузерное уведомление показывается только при наличии разрешения браузера;
- отсутствие разрешения браузера **не отключает** внутренний центр уведомлений.

### 2.5. Поддержка браузерных уведомлений в MVP
В MVP используется только стандартный Browser Notification API в открытом веб-приложении.

В MVP **не** реализуются:
- Web Push;
- Service Worker для фоновой доставки;
- доставка браузерных уведомлений при полностью закрытой вкладке;
- доставка браузерных уведомлений при закрытом браузере.

Итоговое правило:
- если приложение открыто и разрешение выдано, браузер может показать нативное уведомление;
- если вкладка или браузер закрыты, в MVP нативное браузерное уведомление не показывается;
- email-уведомления от состояния вкладки не зависят.

## 3. Типы уведомлений

### 3.1. Итоговый список типов
На экране настроек отображаются ровно 6 типов уведомлений:
1. **Вас добавили в карточку** — `added_to_card`
2. **Сделали ответственным** — `made_responsible`
3. **Новый комментарий в карточке** — `card_comment_new`
4. **Перемещение карточки** — `card_moved`
5. **Ваша карточка в работе** — `card_in_progress`
6. **Ваша карточка готова** — `card_ready`

Других типов на экране настроек нет.

### 3.2. Общее правило “не уведомлять автора”
Если получатель уведомления совпадает с автором действия, уведомление не создаётся ни в одном канале.

Правило применяется ко всем 6 типам уведомлений.

## 4. Матрица событий и получателей

### 4.1. `added_to_card`
Название в UI:
- **Вас добавили в карточку**

Когда создаётся:
- когда пользователь был добавлен в `card_assignees` и до этого не был участником этой карточки.

Получатели:
- только пользователь, которого добавили в карточку.

Если добавление сделал он сам:
- уведомление не создаётся.

### 4.2. `made_responsible`
Название в UI:
- **Сделали ответственным**

Когда создаётся:
- когда `cards.responsible_user_id` меняется с другого значения или `NULL` на конкретного пользователя.

Получатели:
- только новый ответственный пользователь.

Если пользователь сам сделал себя ответственным:
- уведомление не создаётся.

### 4.3. `card_comment_new`
Название в UI:
- **Новый комментарий в карточке**

Когда создаётся:
- когда в карточке создаётся новый комментарий, который не является удалённым.

Получатели:
- все текущие участники карточки из `card_assignees`, кроме автора комментария.

### 4.4. `card_moved`
Название в UI:
- **Перемещение карточки**

Когда создаётся:
- когда карточка перемещена в другую колонку;
- и при этом выполняются одновременно оба условия:
  - тип целевой колонки **не** `in_work`;
  - тип целевой колонки **не** `done`.

Получатели:
- все текущие участники карточки, кроме пользователя, который выполнил перемещение.

Дополнительное правило:
- при обычной смене позиции внутри той же колонки уведомление не создаётся;
- для одного перемещения карточки создаётся только одно уведомление из трёх:
  - `card_moved`
  - `card_in_progress`
  - `card_ready`

### 4.5. `card_in_progress`
Название в UI:
- **Ваша карточка в работе**

Когда создаётся:
- когда карточка перемещена в колонку с `column_type = 'in_work'`;
- и при этом исходная колонка имела другой `column_type`.

Получатели:
- все текущие участники карточки, кроме пользователя, который выполнил перемещение.

Дополнительное правило:
- если карточку переместили между двумя разными колонками, у которых тип уже `in_work`, событие `card_in_progress` не создаётся;
- в таком случае создаётся `card_moved`, если карточка действительно перешла в другую колонку.

### 4.6. `card_ready`
Название в UI:
- **Ваша карточка готова**

Когда создаётся:
- когда карточка перемещена в колонку с `column_type = 'done'`;
- и при этом исходная колонка имела другой `column_type`.

Получатели:
- все текущие участники карточки, кроме пользователя, который выполнил перемещение.

Дополнительное правило:
- если карточку переместили между двумя разными колонками, у которых тип уже `done`, событие `card_ready` не создаётся;
- в таком случае создаётся `card_moved`, если карточка действительно перешла в другую колонку.

## 5. Приоритет событий при перемещении карточки
Для одного факта межколоночного перемещения карточки допускается создание **ровно одного** уведомления по этому набору правил:

1. Если целевая колонка имеет тип `done`, а исходная колонка имела другой тип, создаётся только `card_ready`.
2. Иначе если целевая колонка имеет тип `in_work`, а исходная колонка имела другой тип, создаётся только `card_in_progress`.
3. Иначе если карточка перемещена в другую колонку, создаётся только `card_moved`.
4. Если карточка только поменяла позицию внутри той же колонки, уведомление о перемещении не создаётся.

Это правило обязательно, чтобы пользователь не получал дублирующие уведомления за одно действие.

## 6. Экран настроек уведомлений

### 6.1. Состав экрана
Экран настроек уведомлений содержит только следующие элементы:
- заголовок экрана;
- блок статуса браузерного разрешения;
- кнопку **“Включить уведомления в браузере”** в разрешённом для этого состоянии;
- информационную плашку с правилом “Вы не получаете уведомления, если являетесь автором действия”;
- таблицу настроек уведомлений;
- авто-сохранение изменений без отдельной кнопки “Сохранить”.

На экране **не должно быть**:
- выбора временной зоны;
- блока Telegram;
- настроек тихих часов;
- дополнительных каналов кроме `browser` и `email`.

### 6.2. Таблица настроек
Таблица настроек имеет фиксированную структуру:
- строки — 6 типов уведомлений из раздела 3.1;
- столбец 1 — название типа уведомления;
- столбец 2 — галочка **“В браузере”**;
- столбец 3 — галочка **“По email”**.

Тумблеры не используются.
Используются именно чекбоксы.

### 6.3. Значения по умолчанию
При первом открытии настроек, если у пользователя ещё нет явных пользовательских записей предпочтений:
- для всех 6 типов уведомлений галочка `browser` считается включённой;
- для всех 6 типов уведомлений галочка `email` считается включённой.

### 6.4. Сохранение
Изменение любой галочки сохраняется сразу, без отдельной кнопки подтверждения.

Одно действие пользователя меняет только одну настройку:
- один тип события;
- один канал.

## 7. Блок разрешения браузерных уведомлений

### 7.1. Назначение блока
Блок разрешения управляет только доступом браузера к показу нативных системных уведомлений.

Блок разрешения:
- не заменяет таблицу настроек;
- не включает автоматически email;
- не меняет сами пользовательские чекбоксы предпочтений;
- не отключает внутренний центр уведомлений.

### 7.2. Состояния разрешения
Используются три состояния, соответствующие `Notification.permission`:
- `default`
- `granted`
- `denied`

### 7.3. Поведение кнопки
Если состояние равно `default`:
- показывается активная кнопка **“Включить уведомления в браузере”**;
- нажатие на кнопку вызывает системный запрос браузера на разрешение уведомлений.

Если состояние равно `granted`:
- кнопка не показывается;
- отображается статус: **“Браузерные уведомления включены”**.

Если состояние равно `denied`:
- кнопка не показывается;
- отображается статус: **“Браузер запретил уведомления. Разрешите их вручную в настройках браузера и обновите страницу.”**

### 7.4. Результат после запроса разрешения
После ответа браузера UI обязан немедленно отобразить итоговое состояние:
- `granted` — успешное включение;
- `denied` — запрет;
- `default` — пользователь закрыл системный диалог без выбора, если браузер возвращает это состояние.

## 8. Правила показа нативных браузерных уведомлений

### 8.1. Условия показа
Нативное браузерное уведомление показывается только если одновременно выполняются все условия:
1. для данного типа события у пользователя включена галочка `browser`;
2. браузерное разрешение имеет состояние `granted`;
3. событие прошло правило “не уведомлять автора”;
4. в системе создано соответствующее внутреннее уведомление;
5. вкладка приложения открыта.

### 8.2. Когда нативное уведомление не показывать
Нативное браузерное уведомление не показывается, если выполняется хотя бы одно из условий:
- галочка `browser` выключена;
- разрешение не выдано;
- вкладка приложения закрыта;
- браузер закрыт;
- получатель является автором действия.

### 8.3. Поведение при активной вкладке
Если приложение открыто на активной видимой вкладке:
- запись во внутреннем центре уведомлений создаётся;
- нативное всплывающее уведомление не показывается.

### 8.4. Поведение при неактивной, но открытой вкладке
Если вкладка приложения открыта, но не является активной или видимой:
- запись во внутреннем центре уведомлений создаётся;
- при `Notification.permission = 'granted'` показывается нативное браузерное уведомление.

## 9. Правила email-уведомлений

### 9.1. Когда email создаётся
Email-уведомление создаётся только если одновременно выполняются все условия:
1. для данного типа события у пользователя включена галочка `email`;
2. событие прошло правило “не уведомлять автора”.

### 9.2. Доставка email
Email-канал работает асинхронно через outbox.

Статусы доставки:
- `pending`
- `sent`
- `failed`

Повторные попытки:
- до 5 попыток на одно письмо.

### 9.3. Зависимость от браузера
Email-уведомления не зависят:
- от разрешения браузера;
- от открытой вкладки;
- от состояния окна;
- от активности пользователя в интерфейсе.

## 10. Тексты уведомлений

### 10.1. Заголовки
Фиксированные заголовки для всех каналов:
- `added_to_card` — **Вас добавили в карточку**
- `made_responsible` — **Сделали ответственным**
- `card_comment_new` — **Новый комментарий в карточке**
- `card_moved` — **Перемещение карточки**
- `card_in_progress` — **Ваша карточка в работе**
- `card_ready` — **Ваша карточка готова**

### 10.2. Обязательные данные в теле уведомления
Каждое уведомление должно содержать:
- название доски;
- название карточки, если событие относится к карточке;
- имя автора действия, если автор есть;
- человекочитаемое описание события;
- ссылку, открывающую нужную доску и карточку в приложении.

## 11. Изменения в модели данных

### 11.1. `notification_preferences`
Таблица хранит настройки пользователя по схеме:
- один пользователь;
- один канал;
- один тип события;
- одно boolean-значение `enabled`.

Допустимые значения `channel`:
- `browser`
- `email`

Допустимые значения `event_type`:
- `added_to_card`
- `made_responsible`
- `card_comment_new`
- `card_moved`
- `card_in_progress`
- `card_ready`

### 11.2. `notification_outbox`
Для email-очереди допустимы те же 6 значений `event_type`:
- `added_to_card`
- `made_responsible`
- `card_comment_new`
- `card_moved`
- `card_in_progress`
- `card_ready`

Канал в `notification_outbox` допускается только один:
- `email`

### 11.3. `internal_notifications`
Для внутреннего центра уведомлений допустимы те же 6 значений `event_type`:
- `added_to_card`
- `made_responsible`
- `card_comment_new`
- `card_moved`
- `card_in_progress`
- `card_ready`

### 11.4. `notification_user_settings`
Таблица `notification_user_settings` в рамках этой спецификации больше не требуется.

Причина:
- выбор временной зоны удалён;
- тихие часы удалены;
- разрешение браузера хранится самим браузером, а не в БД.

Итоговое решение:
- таблица удаляется из целевой модели данных;
- приложение не должно зависеть от неё в UI, серверной логике и доставке уведомлений.

## 12. Правила миграции существующих данных

### 12.1. Существующие настройки по старым типам
Если у пользователя уже есть сохранённые настройки по типам:
- `added_to_card`
- `made_responsible`
- `card_comment_new`
- `card_moved`

то эти настройки должны быть сохранены без изменения значений.

### 12.2. Новые типы
Для новых типов:
- `card_in_progress`
- `card_ready`

если у пользователя нет явной записи, значением по умолчанию считается:
- `browser = true`
- `email = true`

### 12.3. Старые пользовательские настройки временной зоны и тихих часов
Старые значения временной зоны и тихих часов:
- не показываются в UI;
- не участвуют в логике доставки;
- не влияют на расчёт времени отправки;
- считаются устаревшими данными.

## 13. Критерии приёмки

### 13.1. Экран настроек
Система считается соответствующей спецификации, если:
- на экране настроек уведомлений нет временной зоны;
- на экране настроек уведомлений нет Telegram;
- в таблице ровно 6 строк типов уведомлений;
- в таблице есть ровно 2 управляемых канала: `В браузере` и `По email`;
- для каждого типа доступны отдельные чекбоксы по обоим каналам.

### 13.2. Кнопка браузерного разрешения
Система считается соответствующей спецификации, если:
- при состоянии `default` видна кнопка **“Включить уведомления в браузере”**;
- при нажатии на кнопку браузер показывает системный запрос разрешения;
- после ответа пользователя экран сразу отражает состояние `granted` или `denied`.

### 13.3. Новые типы уведомлений
Система считается соответствующей спецификации, если:
- при переводе карточки в колонку типа `in_work` создаётся уведомление **“Ваша карточка в работе”**;
- при переводе карточки в колонку типа `done` создаётся уведомление **“Ваша карточка готова”**;
- за одно перемещение карточки создаётся только один тип уведомления из набора `card_moved` / `card_in_progress` / `card_ready`.

### 13.4. Поведение каналов
Система считается соответствующей спецификации, если:
- отключение `email` по типу события прекращает создание email для этого типа;
- отключение `browser` по типу события прекращает создание записей во внутреннем центре уведомлений для этого типа;
- при включённом `browser` и выданном разрешении нативное браузерное уведомление показывается только для открытой, но неактивной вкладки;
- при закрытой вкладке нативное браузерное уведомление не показывается;
- правило “не уведомлять автора” выполняется для всех типов и обоих каналов.

## 14. Что не входит в эту задачу
В рамках этой спецификации не реализуются:
- Telegram-уведомления;
- тихие часы;
