# Спецификация веб‑приложения “Doit” (Trello‑like)

## 1) Цель и границы

### 1.1. Цель
Создать веб‑приложение в стиле Trello для работы с досками, колонками и карточками, с совместной работой, ролями/правами, историей действий и уведомлениями в браузере и по email.

### 1.2. Входит в MVP
- Аутентификация через **Google** и через **email+пароль**.
- Список досок пользователя (в которые он добавлен).
- Доска: колонки, карточки, перемещение и порядок, модальные окна создания/просмотра карточки.
- Администрирование на уровне доски: приглашение участников по email, настройка фона доски, управление ролями участников, настройка пользовательских полей карточек.
- Карточка: описание, участники, метки, комментарии, пользовательские поля, история.
- Уведомления: браузерные (центр уведомлений в приложении) и email по событиям в карточках.
- База данных и реал‑тайм синхронизация: **Supabase (PostgreSQL + RLS + Realtime + Storage + Edge Functions)**.

### 1.3. Не входит в MVP (явно)
- Мобильные приложения (iOS/Android) — только веб.
- Офлайн‑режим.
- Глобальная “организация/команда” поверх досок.
- Импорт/экспорт данных.
- Telegram‑уведомления (перенесены в следующие версии).

## 2) Термины и определения
- **Пользователь**: запись в `auth.users` Supabase + профиль приложения.
- **Доска (Board)**: рабочее пространство с колонками и карточками.
- **Колонка (Column)**: вертикальный список карточек в рамках доски.
- **Тип колонки**: семантическая категория колонки (не является названием). Возможные типы: очередь, в работе, готово, информационный.
- **Карточка (Card)**: элемент работы в колонке; содержит поля, участников, метки, комментарии, историю.
- **Ответственный**: пользователь, который считается текущим ответственным за карточку. В MVP ответственный назначается автоматически при перемещении карточки в колонку типа “В работе” (см. 14.4.4).
- **Участник доски**: пользователь, добавленный в доску (членство).
- **Роль (Role)**: набор прав (permissions) для участника **в рамках конкретной доски**.
- **Системный администратор (System Admin)**: супер‑роль, задается **в коде** (см. 6.6), имеет полный доступ ко всем доскам.
- **Пользовательские поля**: поля карточки, конфигурируемые администратором доски (схема полей на доске).

## 3) Технологические решения (фиксированные)

### 3.1. Хранилище и backend
- **Supabase Postgres** — основная БД.
- **Supabase Auth** — аутентификация.
- **RLS** (Row Level Security) — обязательна для всех таблиц с пользовательскими данными.
- **Supabase Realtime** — обновления доски в реальном времени (колонки/карточки/комментарии/метки/участники/настройки/история).
- **Supabase Storage** — хранение фоновых изображений досок.
- **Supabase Edge Functions** — обработка email outbox/доставки уведомлений.

### 3.2. Аутентификация провайдеров
- **Google**: стандартный OAuth‑провайдер Supabase.
- **Email+пароль**: стандартная аутентификация Supabase Auth.

### 3.3. Идентичность пользователя
- Главный идентификатор пользователя — `auth.users.id` (UUID).
- Email берется из Supabase Auth и считается уникальным для пользователей приложения.

## 4) Сценарии (User Journeys)

### 4.1. Вход / Регистрация
**Экран “Вход”**:
- Варианты входа:
  - Кнопка “Войти через Google”.
  - Форма “Войти по email и паролю”: поля `email`, `password`, кнопка “Войти”.
- Ссылка “Забыли пароль?”:
  - Поле `email`, кнопка “Отправить ссылку для сброса”.
- Ссылка “Создать аккаунт” (экран/вкладка “Регистрация”):
  - Поля: `email`, `password`, `password_repeat`, кнопка “Зарегистрироваться”.
- После успешного входа/регистрации: если нет профиля в приложении — создается профиль.

Ограничения:
- Email обязателен.
- Для входа по email+паролю требуется подтвержденный email (пока email не подтвержден, показывается состояние “Подтвердите email по ссылке из письма”).
- Требования к паролю при регистрации:
  - длина минимум 4 символа (любые символы)

### 4.2. Настройки уведомлений пользователя
**Экран “Настройки профиля”** (доступен всем):
- Блок “Уведомления”.
- Пользователь выбирает временную зону.
- Пользователь настраивает каналы `Браузерные` и `Email` по типам событий.
- Сохранение настроек выполняется сразу при переключении.

### 4.3. Список досок
После входа пользователь попадает на **экран “Доски”**:
- Список карточек‑досок: название, аватар/иконка, количество участников, последнее обновление.
- Отображаются **только доски**, где пользователь является участником.
- Клик по доске открывает экран доски.

### 4.4. Экран “Доска”
Верхняя панель:
- Слева: название доски (клик — переименование, если есть право).
- Справа от названия: кружки‑аватарки всех участников (до N, далее “+X”).
- Кнопка “+” (приглашение участника по email) — видна только при наличии права.

Рабочая область:
- Горизонтальный список колонок.
- Вверху списка колонок: кнопка “+ Колонка” (создать колонку) — по праву.
- В каждой колонке:
  - Заголовок колонки:
    - Название колонки (редактируемое по праву).
    - Бейдж “тип колонки” (одно из: Очередь / В работе / Готово / Информационный). Изменяется пользователем, который может редактировать название колонки.
  - Счетчик карточек (количество карточек в колонке).
  - Кнопка “+” справа от заголовка — открывает модалку создания карточки в этой колонке.
  - Список карточек (клик открывает модалку карточки).
  - Отображение карточки в списке (превью) настраивается администратором доски: какие элементы показывать и в каком порядке (см. 14.5).
  - Перетаскивание карточек внутри колонки и между колонками (по праву).
- Перетаскивание колонок (изменение порядка) — по праву.

Фон доски:
- Может быть цветом или изображением.
- Меняется только администратором доски (по праву).

### 4.5. Создание карточки (модальное окно)
Открывается по кнопке “+” в колонке.
- Модалка по центру экрана, поверх доски.
- Фон затемняется; клик по затемнению закрывает модалку **без сохранения**.
- Поля:
  - “Название” (обязательное)
  - “Участники” (обязательное): минимум 1 участник. По умолчанию выбран создатель карточки.
  - Далее — **пользовательские поля доски** (см. 7)
- Кнопки:
  - “Сохранить” — создает карточку
  - “✕” — закрыть без сохранения

Валидация:
- Название не пустое, длина 1..200.
- “Участники”: выбран минимум 1 пользователь.

### 4.6. Просмотр/редактирование карточки (модальное окно)
Открывается по клику на карточку.
Содержимое:
- Заголовок модалки:
  - В шапке модалки вместо статичного текста “Карточка” показывается текущее название карточки.
  - По клику на заголовок (при наличии права редактирования карточки) заголовок переходит в режим инлайн‑редактирования (input в шапке).
  - Завершение редактирования:
    - `Enter` — принять введённое значение;
    - потеря фокуса (`blur`) — принять введённое значение;
    - `Escape` — отменить изменения заголовка и вернуть исходное название.
- Строка метаданных:
  - “Создана: <дата/время> <создатель> в колонке <название колонки> <время в колонке>”
  - “время в колонке” считается от момента последнего перемещения в эту колонку.
- Участники карточки (обязательный блок отображения; добавление/удаление по праву):
  - Отображаются аватары/чипы всех участников карточки.
  - Рядом отображается кнопка “+” для добавления участников (видна только при наличии прав).
  - Нажатие “+” открывает панель выбора пользователей из участников доски (поиск по имени/email). Выбранные пользователи добавляются в участники карточки.
- Ответственный:
  - Отображается как отдельная метка/поле “Ответственный: <пользователь>”.
  - Может быть пустым, если карточка никогда не перемещалась в колонку типа “В работе”.
  - Может изменяться вручную через панель участника (см. 4.6.1) и автоматически по правилу 14.4.4.
- Метки (добавление/удаление по праву):
  - Отображаются как разноцветные “облачка” (chips) с текстом метки, цвет берется из справочника меток доски.
  - Добавление метки выполняется через поле “Метки” с выпадающим списком (autocomplete) всех меток доски, как на скриншоте:
    - по клику на поле “Метки” выпадающий список открывается даже без ввода текста,
    - ввод фильтрует список по названию,
    - клик по метке в списке добавляет ее на карточку,
    - выбранные метки отображаются отдельными “облачками”.
  - Удаление метки:
    - у каждого “облачка” уже добавленной метки справа отображается крестик “×” (только при наличии прав),
    - клик по “×” удаляет метку с карточки.
- Описание (текст, редактируемое по праву).
- Комментарии (создание/редактирование/удаление по правам).
- Пользовательские поля (отображение и редактирование по праву).
- Кнопка/вкладка “История” (см. 8).

#### 4.6.2. Панель комментариев справа (как на скрине)
В модалке карточки комментарии отображаются **в правой панели** (sidebar), а основная информация карточки — в левой части.
- На desktop (и других широких экранах) используется фиксированная пропорция ширины: **левая часть : правая панель комментариев = 2:1**.
- Правая панель комментариев должна оставаться уже левой части и не расширяться сверх этой пропорции.

Правая панель “Комментарии” содержит:
- Заголовок “Комментарии”.
- Поле ввода нового комментария (вверху панели) + кнопка “Отправить”.
- Лента комментариев сверху вниз (новые сверху).
- Для каждого комментария отображается:
  - автор (аватар + имя),
  - дата/время,
  - текст комментария.
- Для каждого комментария есть только одна кнопка действия: **“Ответить”** (кнопки “Цитировать” в MVP нет).

Поведение кнопки “Ответить”:
- Нажатие “Ответить” переводит фокус в поле ввода нового комментария и устанавливает “контекст ответа” на выбранный комментарий.
- Отправленный комментарий создается как **ответ** на выбранный комментарий:
  - в данных комментария сохраняется ссылка `reply_to_comment_id` на исходный комментарий;
  - в UI ответ визуально помечается строкой “Ответ на комментарий <имя автора>” и/или отображением краткого превью (1 строка) исходного текста.
- Сброс контекста ответа: кнопка “✕” рядом с превью исходного комментария в поле ввода.

Дополнительные поля администратором:
- В модалке карточки есть кнопка “+ Поле” (видна только администратору доски).
- Добавление поля создаёт **новое поле на доске** (board‑scoped), после чего оно доступно на всех карточках этой доски (значения по умолчанию пустые).

#### 4.6.1. Панель участника (popover) в модалке карточки
При нажатии на аватар/чип участника в блоке “Участники” открывается popover‑панель с данными и действиями.

Содержимое панели:
- Аватар.
- Отображаемое имя (ник/имя).
- Email.

Действия панели:
- “Исключить из карточки”:
  - доступно пользователю, который имеет право редактировать участников карточки (см. 5.2 и правило ниже).
  - запрещено, если это приведет к 0 участникам (в карточке всегда минимум 1 участник).
  - если исключаемый пользователь является ответственным, то при исключении:
    - `cards.responsible_user_id` устанавливается в `NULL`;
    - в историю добавляется событие “Снят ответственный: <пользователь>”.
- “Сделать ответственным”:
  - показывается только если выбранный участник **не является** текущим ответственным.
  - доступно пользователю, который имеет право назначать ответственного (см. правило ниже).
  - действие устанавливает `cards.responsible_user_id` в выбранного участника и логируется как “Назначен ответственный: <пользователь>”.

Правило прав для действий панели:
- Действия “Исключить из карточки” и “Сделать ответственным” разрешены если:
  - пользователь имеет `cards.edit_any`, или
  - пользователь имеет `cards.edit_own` и является создателем карточки (`cards.created_by_user_id = auth.uid()`).

Правило прав для кнопки “+” (добавление участников карточки):
- Добавление участников карточки разрешено по тем же правилам, что и действия панели участника (см. выше).

Правила добавления участников:
- Добавить можно только пользователя, который является участником доски (присутствует в `board_members` для этой доски).
- Добавление уже существующего участника карточки не должно создавать дубликат.
- При добавлении участника в `card_activity` добавляется событие “Добавлен участник: <пользователь>”.

## 5) Права доступа: принципы

### 5.1. Общие правила
- Все действия пользователя проверяются через RLS в БД.
- UI скрывает недоступные действия, но основная защита — серверная (RLS).
- Права определяются ролью пользователя **в конкретной доске**.
- Системный администратор имеет полный доступ независимо от членства.

### 5.2. Набор проверяемых возможностей (permissions)
Используется фиксированный набор прав (булевы флаги):
- `board.view`
- `board.rename`
- `board.change_background`
- `board.invite_members`
- `board.remove_members`
- `roles.manage` (назначать роли участникам)
- `columns.create`
- `columns.rename`
- `columns.reorder`
- `columns.delete`
- `cards.create`
- `cards.edit_own`
- `cards.edit_any`
- `cards.move`
- `cards.delete_own`
- `cards.delete_any`
- `card_fields.manage` (создавать/редактировать/удалять пользовательские поля доски)
- `labels.manage` (создавать/редактировать/удалять метки доски и их цвета)
- `card_preview.manage` (настраивать отображение карточки в колонке для доски)
- `comments.create`
- `comments.edit_own`
- `comments.delete_own`
- `comments.moderate` (удалять чужие комментарии)

## 6) Роли

### 6.1. Где назначается роль
Роль назначается **в членстве доски** (один пользователь может иметь разные роли на разных досках).

### 6.2. Роли по умолчанию (фиксировано)
В системе существуют как минимум четыре предустановленные роли (пресеты), создаваемые автоматически для каждой доски:
- **Только просмотр (`viewer`)**: доступ на чтение доски и карточек, без изменений.
- **Редактор (`editor`)**: полноценная работа с карточками/колонками без административных прав.
- **Базовая (`basic`)**: роль по умолчанию для новых участников (по смыслу равна `editor` в MVP, см. 6.4).
- **Администратор доски (`board_admin`)**: управляет доской, участниками, ролями, схемой полей, фоном.

Ключи (`key`) фиксированы: `viewer`, `editor`, `basic`, `board_admin`.

### 6.3. Создание/изменение ролей
- Администратор доски может **создавать дополнительные роли** и настраивать флаги permissions (в рамках доски).
- В каждой доске существует ровно один “владелец” (Owner) — это членство, созданное при создании доски. Owner всегда имеет роль `board_admin` и не может быть лишен доступа к доске.

### 6.4. Назначение ролей
- При добавлении пользователя в доску (после принятия инвайта) ему назначается роль `basic`.
- В MVP роль `basic` функционально эквивалентна роли `editor` (идентичный набор permissions). Разница только в названии (для UX) и возможности администратору позже переопределить `basic` как отдельный пресет.
- Администратор доски может изменить роль любого участника, кроме Owner.

### 6.5. Администратор доски: обязательные права
Роль `board_admin` всегда включает все permissions из раздела 5.2.

### 6.6. Матрица прав для пресетов (фиксировано для MVP)
Ниже перечислены permissions, которые включены (`true`) для каждой предустановленной роли. Все не перечисленные — `false`.

`viewer`:

`basic` (в MVP = `editor`):

### 6.7. Системный администратор (в коде)
- Список системных администраторов задается в конфигурации приложения (например, список email или user_id).
- Системный администратор:
  - Может просматривать/редактировать любые доски.
  - Может назначать себя участником доски при необходимости (опционально).

## 7) Пользовательские поля (схема полей карточек)

### 7.1. Модель
Поля определяются **на уровне доски** и применяются ко всем карточкам доски.

### 7.2. Типы полей (фиксированный перечень)
- **Гиперссылка**: хранится URL + опциональный текст.
- **Текст**: строка/текст.
- **Дата**: дата (без времени).
- **Выпадающий список**: выбор одного значения из заранее заданных опций.

### 7.3. Атрибуты определения поля
Каждое поле имеет:
- Название (1..50).
- Тип (из 7.2).
- Признак обязательности (boolean).
- Порядок отображения (position).
- Для типа “выпадающий список” — список опций (название + цвет + порядок).

### 7.4. Правила изменения схемы
- Администратор доски может:
  - создавать поля,
  - переименовывать,
  - менять порядок,
  - изменять обязательность,
  - удалять поле.
- При удалении поля:
  - значения на карточках удаляются;
  - в истории фиксируется событие “Удалено поле <name>”.

## 8) История (Audit log)

### 8.1. Что фиксируется
Для каждой карточки фиксируются все действия в формате:
`<дата> <время> — <пользователь> — <действие>`

Минимальный список событий:
- Создание карточки.
- Переименование карточки.
- Перемещение карточки (из колонки A в колонку B).
- Автоназначение ответственного (при перемещении в колонку типа “В работе”).
- Ручное назначение ответственного через панель участника.
- Снятие ответственного при исключении ответственного из карточки.
- Изменение описания.
- Добавление/удаление участника карточки.
- Добавление/удаление метки.
- Добавление комментария / удаление комментария / редактирование комментария.
- Изменение пользовательских полей (старое/новое значение).
- Создание/удаление пользовательского поля (на уровне доски).

### 8.2. Отображение
В модалке карточки вкладка “История” показывает ленту событий сверху вниз (новые сверху).
Формат строки, пример:
`25.09.2025 18:10 — Иван Иванов — Перемещена в колонку “Готово” (тип: готово)`

## 9) Приглашения на доску

### 9.1. Добавление по email
На доске администратор нажимает “+” и вводит email.
Система создает **инвайт** в статусе `pending`.

### 9.2. Принятие инвайта
Инвайт принимается автоматически, когда пользователь с этим email впервые:
- входит в приложение, или
- открывает список досок после входа (фоновая проверка).

После принятия:
- создается членство в доске,
- назначается роль `basic`,
- инвайт получает статус `accepted`.

### 9.3. Повторные приглашения
- На одну доску для одного email допускается максимум один активный `pending` инвайт.
- Повторное приглашение возможно только после `cancelled` или `accepted`.

## 10) Уведомления (браузерные + email)

### 10.0. Варианты уведомлений (для удобства пользователя)
Возможные “удобные” схемы уведомлений (в MVP реализуется вариант A):

Вариант A — “Только важное, без шума” (выбран для MVP):
- уведомлять только по событиям, где пользователь явно вовлечён (назначение в карточку, @упоминание, комментарии/перемещения по карточкам, где он участник).

Вариант B — “Всё по доске”:
- уведомлять обо всех изменениях на доске (сильно шумит на активных досках).

Вариант C — “Подписки (watch)”:
- пользователь вручную подписывается на отдельные карточки/колонки/доски и получает события только по подпискам.

Вариант D — “Дайджест”:
- уведомления не моментальные, а сводкой (например, раз в день/час).

Примечание: варианты B–D не входят в MVP.

### 10.1. Каналы уведомлений (фиксировано для MVP)
В MVP поддерживаются 2 канала:
- `browser` — уведомления в приложении (центр уведомлений).
- `email` — письма на адрес пользователя.

Каналы `telegram`, `slack`, `mobile_push` в MVP отсутствуют и в UI не отображаются.

### 10.2. События, которые вызывают уведомления (фиксировано)
Уведомления отправляются пользователям с учетом их настроек (см. 10.6), при событиях:
- Пользователя добавили в участники карточки.
- Пользователя **упомянули** в комментарии карточки (формат упоминания: `@<email>`; матч по `profiles.email`).
- Новый комментарий в карточке:
  - уведомление получают **все текущие участники карточки** (все записи в `card_assignees`), кроме автора комментария.
- Карточка перемещена в другую колонку:
  - уведомление получают **все текущие участники карточки**, кроме пользователя, выполнившего перемещение.
- Пользователя добавили в участники доски (принят инвайт) — уведомление “Вас добавили в доску <name>”.

Примечание (фиксировано для MVP):
- Изменения названия/описания/пользовательских полей карточки не создают уведомления, чтобы не повышать шум.

### 10.3. Формат уведомления
Уведомление (для browser/email) содержит:
- Название доски.
- Название карточки.
- Тип события и краткое описание (кто сделал, куда перемещено и т.п.).
- Ссылку на карточку в веб‑приложении (deeplink на страницу доски + открытие карточки).

### 10.4. Надежность доставки
- Канал `browser` создается записью во внутреннем центре уведомлений.
- Канал `email` отправляется асинхронно через “outbox” таблицу в БД.
- Для `email` каждая запись имеет статус: `pending` → `sent` или `failed`.
- Повторные попытки для `email`: до 5 раз с увеличивающейся задержкой.

### 10.6. Настройки уведомлений (как на скрине, фиксировано для MVP)
Настройки хранятся на уровне пользователя (не зависят от доски) и позволяют включать/выключать получение уведомлений по типам событий отдельно для каждого канала.

#### 10.6.1. Типы уведомлений (фиксировано)
В настройках отображаются только следующие типы (ровно 4, как указано):
1) “Вас добавили в карточку” (`added_to_card`)
2) “Сделали ответственным” (`made_responsible`)
3) “Новый комментарий в карточке” (`card_comment_new`)
4) “Перемещение карточки” (`card_moved`)

#### 10.6.2. Правило “не уведомлять, если вы автор действия”
Пользователь **не получает** уведомления по событию, если он является автором действия (actor) этого события. Правило применяется ко всем каналам.

#### 10.6.3. UI экрана “Настройка уведомлений”
Экран повторяет структуру как на скрине:
- Выпадающий список “Временная зона”:
  - значение хранится как IANA‑таймзона (например, `Europe/Moscow`)
  - по умолчанию: `Europe/Moscow`
- Информационная плашка: “Вы не получаете уведомления, где являетесь автором действий”.
- Таблица:
  - строки — типы из 10.6.1
  - столбцы — каналы `Email` и `Браузерные` (оба с тумблерами on/off)
- Сохранение происходит мгновенно при переключении (без отдельной кнопки “Сохранить”).

#### 10.6.4. Тихие часы (фиксировано для MVP)
Тихие часы применяются **только к email**.
Настройки:
- `quiet_hours_enabled` (default `false`)
- `quiet_hours_start_local` (time, null)
- `quiet_hours_end_local` (time, null)

Правило:
- если `quiet_hours_enabled=true`, то email‑уведомления не отправляются в интервале [start, end) по локальному времени выбранной таймзоны; такие уведомления остаются “ожидающими отправки” и переносятся на ближайшее время после окончания тихих часов.

## 11.10.1. Управление метками (администратор доски)
Требования:
- Список меток (их названия и цвета) формируется администратором доски.
- Администратор доски может:
  - создавать метки,
  - менять название,
  - менять цвет,
  - менять порядок,
  - удалять метки.

Правило прав:
- Управление каталогом меток доступно если у пользователя `labels.manage` (включено у `board_admin`).

Поведение при удалении метки:
- Удаленная метка автоматически удаляется со всех карточек доски (удаляются соответствующие записи `card_labels`).
- В `card_activity` для каждой затронутой карточки добавляется событие “Удалена метка: <name>”.

## 12) RLS и безопасность (высокоуровневые правила)

### 12.1. Общие принципы RLS
RLS включена на всех таблицах раздела 11, кроме служебных Supabase.
Доступ разрешается если:
- пользователь — системный администратор, или
- пользователь — участник соответствующей доски и действие разрешено его role permissions.

### 12.2. Членство как основа авторизации
Проверка членства выполняется через `board_members`.
Для таблиц карточек/комментариев/истории проверяется `cards.board_id`.

### 12.3. Профили
Пользователь может читать/редактировать только свой `profiles` (кроме read‑полей, которые доступны участникам одной доски для отображения аватарок).

## 13) Реал‑тайм (совместная работа)

### 13.1. Что должно обновляться у всех участников без перезагрузки
- Создание/удаление/переименование колонок, изменение порядка.
- Создание/удаление/перемещение/переименование карточек, изменение порядка.
- Изменения описания, меток, участников карточки.
- Комментарии.
- Схема пользовательских полей и их значения.
- Настройки доски (фон, название).

### 13.2. Конфликт изменений
При одновременном редактировании:
- Последняя запись в БД считается истинной (“last write wins”).
- История фиксирует изменения обеих сторон как отдельные события.

## 14) Требования к UI/UX

### 14.1. Модальные окна
- Открытие/закрытие с затемнением фона.
- Закрытие по ESC.
- Закрытие по клику вне модалки.
- При закрытии без сохранения изменения не сохраняются.

### 14.2. Drag & Drop
- Колонки и карточки перетаскиваются мышью.
- На время перетаскивания показывается placeholder.
- Порядок сохраняется в `position` (double precision).

### 14.3. Счетчики
- В заголовке колонки отображается количество карточек в ней (по данным списка).

## 14.4. Типы колонок (UI/UX и правила)
### 14.4.1. Типы (фиксированный перечень)
- `queue` — отображаемое имя “Очередь”
- `in_work` — отображаемое имя “В работе”
- `done` — отображаемое имя “Готово”
- `info` — отображаемое имя “Информационный”

### 14.4.2. Правила
- Тип колонки **не зависит** от названия колонки и может совпадать/не совпадать по смыслу с названием.
- В одной доске может быть несколько колонок одного типа.
- Карточки могут находиться в колонках любого типа; специальных ограничений на перемещение по типам в MVP нет.

### 14.4.3. Значения по умолчанию при создании доски (фиксировано)
При создании новой доски автоматически создаются 4 колонки (слева направо):
1) `name="Очередь"`, `column_type="queue"`
2) `name="В работе"`, `column_type="in_work"`
3) `name="Готово"`, `column_type="done"`
4) `name="Информационный"`, `column_type="info"`

### 14.4.4. Автоназначение “Ответственного” (фиксировано для MVP)
Правило:
- Когда карточка перемещается **в колонку** с `column_type="in_work"`, пользователь, совершивший перемещение, становится **Ответственным** по карточке:
  - `cards.responsible_user_id` устанавливается в `actor_user_id` перемещения.
  - Если пользователь не был участником карточки, он добавляется в `card_assignees`.
  - В `card_activity` добавляется событие “Назначен ответственный: <пользователь>” (вместе с событием перемещения или отдельным событием, но оба должны присутствовать).

Повторные перемещения:
- При каждом новом перемещении карточки в колонку типа “В работе” ответственный перезаписывается на нового переносившего.

Перемещения из “В работе”:
- При перемещении карточки **из** колонки типа “В работе” ответственный **не сбрасывается** и остается последним назначенным.

Приоритет автоназначения:
- Если ответственный был назначен вручную, последующее перемещение карточки в колонку типа “В работе” **перезаписывает** ответственного на пользователя, выполнившего перемещение.

## 14.5. Отображение карточки в колонке (превью) — настройка администратором

### 14.5.1. Цель
В колонках (на доске) карточка отображается компактно. Администратор доски настраивает, какие элементы показывать на карточке в колонке и в каком порядке — как на предоставленном примере.

### 14.5.2. Доступ
Настройка доступна только пользователю, у которого есть permission `card_preview.manage` (в MVP — у `board_admin`).

### 14.5.3. Где находится настройка
В меню настроек доски: раздел “Отображение карточек в колонках”.

### 14.5.4. Доступные элементы (фиксированный список)
Администратор может включать/выключать и менять порядок следующих элементов:
- `title` — заголовок карточки (всегда отображается первым и не может быть отключен).
- `assignees` — участники карточки (аватары/инициалы).
- `comments_count` — счетчик комментариев (иконка + число, считаются только комментарии без `deleted_at`).
- `labels` — метки (цветные “облачка”).
- `responsible` — ответственный (бейдж “Ответственный” + аватар/инициалы).
- `custom_field` — любое пользовательское поле доски (из `board_field_definitions`).

Правило для `custom_field`:
- Администратор выбирает конкретные пользовательские поля, которые будут показаны на карточке в колонке.
- Для каждого такого поля создается отдельный элемент `board_card_preview_items` с `item_type="custom_field"` и соответствующим `field_definition_id`.

### 14.5.5. Правила отображения значений
- `title`: 1–2 строки, с обрезкой по ширине колонки.
- `assignees`: ряд аватаров/инициалов (до 5), далее “+N”.
- `comments_count`: число рядом с иконкой комментария.
- `labels`: отображаются как небольшие “облачка”, как в карточке.
- `responsible`: отображается только если `cards.responsible_user_id` не NULL.
- `custom_field`:
  - `text`: показывается первая строка (обрезка по ширине).
  - `date`: показывается в формате локали пользователя (дата без времени).
  - `select`: показывается выбранная опция как “облачко” с цветом опции.
  - `link`: показывается текст ссылки (если задан), иначе домен/URL с обрезкой.

### 14.5.6. Значения по умолчанию (фиксировано)
Для новой доски по умолчанию включены и упорядочены элементы:
1) `title`
2) `assignees`
3) `comments_count`
4) `labels`

Элементы `responsible` и `custom_field` по умолчанию выключены.

## 15) Нефункциональные требования
- Время отклика основных операций (создание/перемещение карточки) — до 300 мс при нормальной сети (без учета загрузки изображений).
- Все даты/время в интерфейсе — локальная таймзона пользователя, хранение в БД — UTC.
- Логи действий (card_activity) не удаляются автоматически.
- Удаление комментариев — мягкое (`deleted_at`) с фиксацией в истории.

### 15.1. Бюджет запросов и стратегия обновлений (performance)
Требование: маленькая доска (пример: 2 пользователя, 5 карточек, минимальная активность) не должна генерировать избыточный трафик к Supabase за счёт “полных перезагрузок данных”.

Целевые ориентиры (не строгие SLA, но критерии для оптимизации):
- Первичная загрузка `/boards/[boardId]`: **1–3 DB запроса**, **≤ 1 Auth запрос**, **0–1 Storage запрос** (только если нужен фон‑изображение).
- Realtime: пачка событий должна приводить максимум к **1 refresh/snapshot** (batched), а при возможности — к точечному обновлению локального состояния без полного snapshot.

Стратегия:
- Предпочитать **агрегированный snapshot** доски (RPC `get_board_snapshot(board_id)` или эквивалентный механизм), чтобы избежать 10–15+ отдельных round‑trip запросов при каждом SSR/refresh.
- Для фона‑изображения доски: signed URL должен **кэшироваться** с TTL (например, 30–60 минут) и пересоздаваться только при изменении `background_image_path`.
- Realtime-обновления не должны триггерить `router.refresh()` на каждое одиночное событие: нужен debounce/батчинг, а где безопасно — применение изменений точечно по payload.
