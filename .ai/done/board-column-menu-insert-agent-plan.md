# План реализации "Создание колонки из меню колонки" (для AI-агента)

Основано на `.ai/board-column-menu-insert-specification.md`.

Цель плана: дать исполняемую декомпозицию для переноса создания колонки из глобальной кнопки в меню конкретной колонки, внедрения вставки строго после исходной колонки и запрета на удаление последней колонки с UI- и server-side защитой.

## 0) Входные решения (зафиксировано спецификацией)
- Глобальная кнопка `+ Колонка` на экране доски должна исчезнуть.
- Единственный entry-point создания колонки на экране доски: пункт `Добавить колонку` в меню конкретной колонки.
- Новая колонка всегда создаётся относительно исходной колонки, а не "в конец по умолчанию".
- Исходная колонка определяется по колонке, из меню которой открыт сценарий создания.
- Модалка создания остаётся той же по сути:
  - заголовок `Новая колонка`;
  - обязательные поля `Название` и `Тип`;
  - `placeholder` названия: `Новая колонка`;
  - тип по умолчанию: `queue`;
  - кнопки `Создать` и `Отмена`.
- В модалке создания обязателен текст:
  - `Новая колонка будет добавлена сразу после колонки "<название исходной колонки>".`
- Серверная операция создания на экране доски обязана получать исходную колонку.
- Если исходная колонка не передана, не найдена или не принадлежит доске, ошибка фиксирована:
  - `Колонка не найдена на этой доске.`
- Вставка рассчитывается по актуальному состоянию БД на момент мутации, а не по клиентскому snapshot.
- Последнюю колонку на доске удалять нельзя ни через UI, ни прямым server-side вызовом.
- Ошибка попытки удалить последнюю колонку фиксирована:
  - `Нельзя удалить последнюю колонку на доске.`
- Для последней колонки пункт `Удалить колонку…` остаётся видимым, но disabled, и под ним показывается пояснение:
  - `Последнюю колонку на доске удалить нельзя.`
- Права `columns.create`, `columns.rename`, `columns.delete` независимы и не должны смешиваться.
- Логика `columns.reorder`, drag-and-drop колонок и карточек не должна меняться.

  - не гарантирует серверный алгоритм "вставить строго после исходной".

### 1.2. Явные расхождения с целевым состоянием
- На экране доски всё ещё существует отдельная кнопка `+ Колонка`.
- Меню колонки не учитывает `columns.create` как самостоятельный повод показать кнопку `⋮`.
- Создание колонки не привязано к исходной колонке.
- Создание колонки всегда идёт в конец, что прямо противоречит спецификации.
- В модалке создания нет обязательного пояснения про исходную колонку.
- Удаление последней колонки не заблокировано на сервере.
- UI не умеет держать `Удалить колонку…` в disabled-состоянии для последней колонки с пояснением.
- При текущем коде право `columns.create` не даёт открыть меню, если нет `rename/delete`.

## 2) Стратегия выполнения
- Сначала убрать архитектурное противоречие в серверной мутации:
  - создание должно стать source-column-based;
  - удаление должно стать last-column-safe.
- Затем перестроить UI header/menu вокруг независимых прав `create/rename/delete`.
- После этого удалить глобальную кнопку, но сохранить высоту control-row.
- Затем закрыть UX-детали:
  - поясняющий текст в модалке;
  - disabled delete для последней колонки;
  - прокрутка к новой колонке после успешного создания.
- В конце пройти acceptance matrix из спецификации по правам, порядку вставки и последней колонке.

Порядок внедрения:
1. Зафиксировать и расширить серверный контракт создания/удаления.
2. Подготовить данные UI для понимания "последняя ли это колонка".
3. Перестроить `BoardColumnHeader` и меню.
4. Переиспользовать/перенести create-modal в сценарий "после этой колонки".
5. Удалить `AddBoardColumnButton` из control-row, не ломая layout.
6. Добавить post-success scroll/focus-поведение.
7. Провести ручную и техническую приёмку.

## 3) Трекер задач (живой чеклист)
Статусы: `todo | doing | blocked | done`.

### EPIC BCM1 - Подготовить серверный контракт вставки "после исходной колонки"
- [x] **BCM1.1 (done)** Выбрать точку реализации вставки после исходной колонки
  - предпочтительно: отдельный RPC/SQL function с транзакционным пересчётом позиций;
  - допустимо: server action, если гарантируется консистентный пересчёт без race-condition;
  - решение должно работать по актуальному состоянию БД на момент запроса.
  - **DoD**: выбран один серверный путь, который не опирается на клиентский snapshot порядка.
  - **Решение зафиксировано**: реализуем через отдельный `SECURITY DEFINER` RPC в новой SQL-миграции для `board_columns`, а `createBoardColumnAction(...)` оставим тонким адаптером над `supabase.rpc(...)`.
  - **Почему так**: в проекте уже есть такой паттерн для многосоставных/атомарных мутаций (`create_card_with_details`, `delete_board_label_with_activity`), а текущий `server action` с отдельными `select/update/insert` не даёт надёжной защиты от race-condition при конкурентных вставках.
  - **Техническая рамка**: RPC сам валидирует `source_column_id`, читает актуальную позицию source-колонки из БД, сдвигает соседние колонки и вставляет новую в одной server-side операции; клиентский snapshot порядка не используется.
- [x] **BCM1.2 (done)** Реализовать обязательную валидацию `source_column_id`
  - источник должен быть передан явно;
  - колонка должна существовать;
  - колонка должна принадлежать переданной доске;
  - при любом нарушении вернуть:
    - `Колонка не найдена на этой доске.`
  - **DoD**: создание без валидной исходной колонки невозможно.
  - **Сделано**: добавлена SQL-миграция `supabase/migrations/20260410143000_create_board_column_after_rpc.sql` с `SECURITY DEFINER` RPC `public.create_board_column_after(...)`; внутри реализована обязательная проверка `p_source_column_id IS NOT NULL` и принадлежности `board_columns.id` нужной доске, при нарушении поднимается фиксированная ошибка `Колонка не найдена на этой доске.`.
- [x] **BCM1.3 (done)** Реализовать алгоритм вставки "строго после исходной"
  - найти актуальную позицию исходной колонки в БД;
  - сдвинуть все колонки справа на `+1`;
  - вставить новую колонку на `source.position + 1`;
  - сохранить относительный порядок остальных колонок.
  - **DoD**: новая колонка всегда появляется сразу после исходной, даже если пользователь держал старую модалку.
  - **Сделано**: в RPC `public.create_board_column_after(...)` заменён сценарий "в конец" на вставку после source-колонки по актуальному состоянию БД: читается `source.position` с `FOR UPDATE`, затем выполняется `UPDATE board_columns SET position = position + 1 WHERE position > source.position`, после чего новая колонка вставляется на `source.position + 1`.
  - **Hotfix (2026-04-10)**: добавлена и применена миграция `supabase/migrations/20260410171000_fix_create_board_column_after_position.sql`, которая принудительно переопределяет RPC `create_board_column_after` с алгоритмом вставки строго после source-колонки (сдвиг всех колонок справа на `+1`).
  - **Проверка/применение**: попытка применить миграции локально (`supabase migration up`) упёрлась в недоступный локальный Postgres; `supabase start` сообщает, что Docker Desktop не запущен/не установлен.
- [x] **BCM1.4 (done)** Обновить `createBoardColumnAction(...)` под новый контракт
  - принимать `sourceColumnId`;
  - больше не вычислять `max(position)+1` как основной сценарий;

    - право `columns.create`.

  - **Сделано**: в `web/src/app/boards/[boardId]/actions.ts` `createBoardColumnAction(...)` теперь требует `source_column_id` из `FormData` (при отсутствии возвращает `Колонка не найдена на этой доске.`) и вызывает RPC `create_board_column_after` вместо прямого `insert` с `max(position)+1`.

### EPIC BCM2 - Защитить удаление последней колонки на сервере
- [x] **BCM2.1 (done)** Выбрать server-side enforcement для "нельзя удалить последнюю колонку"
  - предпочтительно: RPC/trigger/function в БД;
  - важно: правило должно срабатывать не только из UI, но и при прямом вызове мутации.
  - **DoD**: защита живёт ниже UI и не обходится клиентом.
  - **Решение зафиксировано**: реализуем ограничение через `BEFORE DELETE` trigger на `public.board_columns` в новой SQL-миграции; проверка количества колонок на доске и генерация фиксированной ошибки выполняются в trigger function на стороне БД.
  - **Почему так**: trigger покрывает любой путь удаления (UI action, прямой SQL delete, сторонние клиенты), поэтому бизнес-правило невозможно обойти клиентским кодом.
- [x] **BCM2.2 (done)** Встроить приоритет проверок удаления по спецификации
  - 1. авторизация;
  - 2. существование колонки в нужной доске;
  - 3. на доске больше одной колонки;
  - 4. остальные ограничения, включая запрет удаления колонки с карточками.
  - **DoD**: при последней колонке сообщение про last-column имеет приоритет над сообщением про карточки.
  - **Сделано**: добавлена SQL-миграция `supabase/migrations/20260410150000_prevent_last_board_column_delete.sql` с `BEFORE DELETE` trigger `trg_prevent_last_board_column_delete` на `public.board_columns`.
  - **Почему это даёт нужный приоритет**: триггер выполняется до проверки FK `cards -> board_columns`, поэтому при попытке удалить последнюю колонку ошибка last-column возникает раньше кейса "в колонке есть карточки".
- [x] **BCM2.3 (done)** Вернуть фиксированную ошибку при удалении последней колонки
  - текст:
    - `Нельзя удалить последнюю колонку на доске.`
  - **DoD**: любой server-side сценарий удаления последней колонки возвращает именно это сообщение.
  - **Сделано**: в `supabase/migrations/20260410150000_prevent_last_board_column_delete.sql` триггер `trg_prevent_last_board_column_delete` поднимает фиксированную ошибку `Нельзя удалить последнюю колонку на доске.` при попытке удалить последнюю колонку.
  - **Покрытие сценариев**: поскольку правило реализовано в `BEFORE DELETE` trigger на таблице `public.board_columns`, сообщение возвращается для любого server-side delete-сценария, а не только для UI.
- [x] **BCM2.4 (done)** Обновить `deleteBoardColumnAction(...)`
  - правильно маппить server-side ошибку про последнюю колонку;
  - сохранить действующее сообщение для случая, когда в не последней колонке есть карточки.
  - **DoD**: delete action различает "последняя колонка" и "есть карточки".
  - **Сделано**: в `web/src/app/boards/[boardId]/actions.ts` (функция `deleteBoardColumnAction`) добавлен явный маппинг server-side ошибки `Нельзя удалить последнюю колонку на доске.` перед обработкой `23503`, при этом существующее сообщение про карточки для не последней колонки сохранено без изменений.

## 6) Риски и контрольные точки
- Риск: реализация create-after останется на уровне нескольких несвязанных `update/insert` из `server action`, что создаст race-condition.
  - Контроль: предпочесть один транзакционный SQL-сценарий.
- Риск: UI будет определять "последняя колонка" только по устаревшему клиентскому состоянию.
  - Контроль: использовать это лишь для disabled-состояния UI, но не для окончательного запрета; окончательное правило должно жить на сервере.
- Риск: меню колонки по-прежнему будет скрыто для пользователя с одним только `columns.create`.
  - Контроль: отдельно проверить условие `showMenu`.
- Риск: удаление `AddBoardColumnButton` сломает высоту control-row.
  - Контроль: после удаления кнопки проверить `board-canvas.tsx` на сохранение layout-контейнера.
- Риск: create-modal будет переиспользована частично, но сохранит устаревший текст про добавление в конец.
  - Контроль: выполнить прямой поиск по строке `Колонка добавляется в конец единого горизонтального ряда.`
- Риск: после создания колонка появится вне viewport, и пользователь решит, что операция не сработала.
  - Контроль: отдельно реализовать post-success scroll к новой колонке.
- Риск: запрет удаления последней колонки будет реализован только в UI.
  - Контроль: протестировать прямой server-side вызов.
