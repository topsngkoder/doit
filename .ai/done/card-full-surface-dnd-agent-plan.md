# План реализации "Вся поверхность карточки как drag-and-drop" (для AI-агента)

Основано на `.ai/card-full-surface-dnd-specification.md`.

Цель плана: дать AI-агенту исполняемую декомпозицию для внедрения full-surface drag-and-drop карточек на доске без выхода за рамки спецификации: вся поверхность карточки становится единой drag-зоной, ручка `⋮⋮` удаляется, внутренние элементы карточки теряют собственную кликабельность, а клик без начала drag продолжает работать только как общее действие карточки по матрице прав.

## 0) Зафиксированные требования спецификации
- Вся визуальная поверхность карточки в колонке доски является единой drag-зоной.
- Название карточки входит в drag-зону наравне со всеми остальными внутренними элементами.
- Внутри карточки не остаётся исключений: метки, участники, комментарии, responsible, custom fields, ссылки, даты и любые другие превью-элементы не должны перехватывать drag и не должны иметь собственного клика.
- Визуальная ручка `⋮⋮` убирается полностью.
- Механика старта drag должна остаться той же, что сейчас у ручки: тот же activation threshold, тот же сценарий drag/cancel/drop.
- Если drag не начался, обычный клик/тап по карточке работает как текущее общее действие карточки, но только в рамках матрицы прав.
- После drag или cancel карточка не должна открываться автоматически.
- Если у пользователя нет права перемещать карточки, карточка не должна становиться drag-зоной.
- Если у пользователя нет права открывать карточку, обычный клик без drag не должен выполнять дополнительное действие.
- Внутренние элементы карточки не должны быть отдельными focusable/clickable controls.
- Для всей поверхности карточки должны корректно работать курсор, визуальное drag-состояние, запрет нежелательного выделения текста и запрет нативного browser drag внутреннего контента.
- Требования одинаковы для мыши и touch.

## 1) Текущий технический контекст проекта

### 1.1. Что уже есть в коде
- `web/src/app/boards/[boardId]/board-columns-dnd.tsx`
  - основное место реализации карточки в колонке;
  - `BoardCardRow` рендерит всю визуальную поверхность карточки;
  - `SortableBoardCard` подключает `useSortable`;
  - сейчас `listeners`/`attributes` `dnd-kit` передаются только в кнопку-ручку `⋮⋮`;
  - карточка открывается через `onOpen(card)` по `onClick`/`onKeyDown`;
  - `PointerSensor` уже настроен с `activationConstraint: { distance: 8 }`;
  - есть общий DnD lifecycle: `onDragStart`, `onDragCancel`, `onDragEnd`.
- `web/src/app/boards/[boardId]/column-types.ts`
  - содержит `canOpenCardModal(...)`;
  - сейчас "право открыть карточку" вычисляется не отдельным permission, а через текущую продуктовую логику: `edit_any | edit_own | delete_any | delete_own | assignee-edit-body`.
- `web/src/app/boards/[boardId]/page.tsx`
  - источник `canMoveCards` из `allowed_permissions`;
  - сюда уже сведены права и snapshot, которыми питается UI доски.
- `web/src/app/boards/[boardId]/board-canvas.tsx`
  - обёртка над доской, прокидывает состояние и флаги в `BoardColumnsDnD`.
- `web/src/app/boards/[boardId]/edit-card-modal.tsx`
  - открытая карточка; сама по себе не является центром этой доработки, но поведение открытия модалки зависит от клика по карточке на доске.

### 1.2. Что в текущей реализации уже близко к спецификации
- Drag карточек уже реализован через `dnd-kit`.
- Порог старта drag уже существует и не должен меняться.
- Внутри карточки в превью почти нет отдельных интерактивных элементов, кроме drag handle.
- Keyboard/open поведение уже привязано к самому контейнеру карточки, а не к её вложенным элементам.

### 1.3. Что сейчас не соответствует спецификации
- Drag инициируется только через кнопку `⋮⋮`, а не через всю поверхность карточки.
- В UI всё ещё присутствует отдельная визуальная drag-ручка.
- Курсор drag-поведения относится к ручке, а не ко всей карточке.
- Нет явно зафиксированного слоя защиты от ложного открытия карточки после drag со всей поверхности.
- Не зафиксированы защитные меры против text selection и native browser drag для всех внутренних элементов карточки.
- Матрица прав спецификации сформулирована как `move` x `open`, тогда как текущий код опирается на существующую функцию `canOpenCardModal(...)`; это нужно сохранить консистентно и не "улучшать" без явной необходимости.

## 2) Рабочие правила для AI-агента
- Не менять продуктовую семантику открытия карточки шире, чем требует спецификация.
- Не вводить новое permission "open card", если это не требуется существующей архитектурой и не подтверждено пользователем.
- Не добавлять новый клавиатурный drag-flow.
- Не возвращать кликабельность внутренним элементам карточки "заодно".
- Не трогать содержимое модалки карточки, кроме мест, напрямую связанных с открытием карточки из списка.
- Не перестраивать DnD колонок: эта доработка относится только к карточкам.
- Любая спорная трактовка по праву "открыть карточку" или по доступности карточки при текущей модели прав является стоп-условием: остановиться и спросить пользователя, а не принимать архитектурное решение молча.

## 3) Стратегия выполнения
1. Сначала зафиксировать точную матрицу поведения на базе текущих прав и существующего `canOpenCardModal(...)`.
2. Затем перенести card drag activator с отдельной ручки на общий контейнер карточки.
3. После этого удалить ручку `⋮⋮` и привести всю карточку к единому pointer/cursor UX.
4. Затем отдельно добить ложные клики после drag, user-select/native drag и поведение внутренних элементов.
5. После этого выровнять non-DnD рендер, keyboard/focus и все четыре комбинации прав из спецификации.
6. В конце провести ручную приёмку по матрице сценариев и только после этого, при необходимости, добавить/обновить точечные тесты.

Критическое правило порядка:
- Нельзя начинать косметическую чистку классов и мелкий рефакторинг, пока не доказано, что full-surface activation не ломает матрицу `drag vs open` и не даёт ложного открытия после drag.

#### Матрица `canMoveCards` × `canOpenCardModal` (зафиксировано DND1.1)

| `canMoveCards` (`has("cards.move")` в `page.tsx`) | `canOpenCardModal(perms, card, user)` | Ожидаемое поведение по спецификации full-surface DnD |
|--------------------------------------------------|----------------------------------------|------------------------------------------------------|
| да | да | Вся поверхность — drag-зона; короткий клик открывает модалку; после drag/cancel — не открывать. |
| да | нет | Вся поверхность — drag-зона; клик без drag не даёт открытия и лишних действий. |
| нет | да | Нет drag listeners; клик по всей поверхности открывает карточку. |
| нет | нет | Статичная карточка: без role кнопки, без tabIndex ради открытия, без drag, без курсора «рука» как у кликабельного контрола. |

#### Контракт `dnd-kit` для full-surface карточки (DND1.3)

**Где в коде (на 2026-04-10):** `web/src/app/boards/[boardId]/board-columns-dnd.tsx` — `useSensors` / `DndContext`.

| Элемент | Текущее значение | Правило для последующих эпиков (DND2+) |
|---------|------------------|----------------------------------------|
| `PointerSensor` | `activationConstraint: { distance: 8 }` | **Не менять** порог и не подменять на другой сенсор ради карточки. После переноса activator с ручки на карточку тот же порог применяется ко всей поверхности. |
| `KeyboardSensor` | `coordinateGetter: sortableKeyboardCoordinates` | **Не расширять** (нет отдельного keyboard-drag по карточке; см. спецификацию / стоп-условия плана). Менять только по явному продуктовому решению. |
| `DndContext` | `collisionDetection={closestCorners}`, `onDragStart` / `onDragCancel` / `onDragEnd` | Не перестраивать ради этой задачи. |
| `useSortable` (карточка) | `id: card.id`, `disabled: !enableDrag` | Менять только привязку **`attributes` / `listeners`** к DOM (surface), плюс при необходимости guards «клик vs drag» **без** изменения списка сенсоров и `distance`. |

**Итог:** допустимые правки по `@dnd-kit` в рамках full-surface — в основном **смена узла-activator** для `listeners`/`attributes` карточки и сопутствующая логика клика; фундаментальная механика сенсоров и коллизий остаётся как в таблице выше.

## 6) Риски и защита от ошибок
- **Риск: ложное открытие карточки после drag.**
  - Защита: явная ручная проверка `drag start -> drop/cancel -> no open`; при необходимости локальный anti-click guard.
- **Риск: touch начинает конфликтовать со scroll колонки.**
  - Защита: отдельно проверить на реальном touch/эмуляции до завершения задачи.
- **Риск: появится несоответствие между DnD и static/mount fallback.**
  - Защита: использовать единый `BoardCardRow` contract и проверить оба пути рендера.
- **Риск: агент случайно изменит semantics "кто может открыть карточку".**
  - Защита: не менять `canOpenCardModal(...)` без явной необходимости; при сомнении остановиться и спросить пользователя.
- **Риск: вложенные элементы станут визуально интерактивными через hover/title/cursor.**
  - Защита: курсор и focusable-состояния централизовать на контейнере карточки, а не на дочерних узлах.

## 7) Стоп-условия: когда агент обязан остановиться и спросить пользователя
- Если выяснится, что спецификацию невозможно корректно уложить в текущую модель `canOpenCardModal(...)` без введения нового permission или server-side изменения прав.
- Если на touch full-surface drag фундаментально конфликтует со scroll колонки и требуется продуктовый компромисс по gesture behavior.
- Если в других местах проекта найдутся скрытые кликабельные внутренние элементы карточки, не описанные в текущем preview-рендере, и их отключение меняет уже используемый UX.
- Если при удалении ручки `⋮⋮` всплывёт отдельный accessibility-конфликт, требующий нового keyboard drag-сценария.

## 8) Критерий завершения всей задачи
Задача считается завершённой только когда:
- drag стартует из любой визуальной точки карточки;
- ручка `⋮⋮` полностью исчезла;
- внутренние элементы карточки не имеют собственного клика;
- клик без drag работает строго по текущему праву открытия карточки;
- после drag/cancel карточка не открывается;
- все 4 комбинации прав из спецификации дают ровно ожидаемое поведение;
- изменённые файлы проходят lint и ручную приёмку.
