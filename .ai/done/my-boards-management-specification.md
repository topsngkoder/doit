# Спецификация: “Мои доски” — переименование, удаление, “доска по умолчанию”

## 1. Назначение документа
Документ фиксирует однозначные требования к доработке экрана **“Мои доски”** (`/boards`) и к навигации по приложению:
- добавить возможность **переименовывать** доску;
- добавить возможность **удалять** доску;
- добавить флажок **“Доска по умолчанию”** (одна доска на пользователя);
- обеспечить редирект на доску по умолчанию:
  - при заходе на главную страницу приложения (`/`) авторизованным пользователем;
  - при нажатии на кнопку/логотип **DOIT** в верхнем меню.

В спецификации **нет кода**. Все решения конкретны, без двусмысленности.

## 2. Термины
- **Доска**: запись в таблице `boards`.
- **Участник доски**: пользователь, имеющий запись в `board_members` для этой доски.
- **Владелец доски (Owner)**: пользователь, указанный как `boards.owner_user_id` (и имеющий членство `board_members.is_owner = true`).
- **Доска по умолчанию**: доска, выбранная пользователем как целевая для “быстрого входа” в приложение.

## 3. Фиксированные решения

### 3.1. Что означает “удалить доску”
Действие **“Удалить”** означает:
- полное удаление доски из системы;
- каскадное удаление всех связанных данных этой доски (колонки, карточки, метки, история, участники и т. п.);
- удаление влияет на **всех участников** (доска перестаёт существовать).

“Удалиться из доски” (покинуть доску без удаления) в рамках этой задачи **не реализуется**.

### 3.2. Права и ограничения (обязательные)
- **Переименование доски** доступно только пользователям, у которых на этой доске есть право `board.rename` (проверяется в БД через RLS).
- **Удаление доски** доступно только:
  - владельцу доски (`boards.owner_user_id = auth.uid()`), или
  - системному администратору (по правилу `public.is_system_admin()`).
- **Выбор “доски по умолчанию”** доступен любому авторизованному пользователю для любой доски, где он является участником (есть `board.view`).

UI может скрывать недоступные действия, но окончательное решение — **на стороне БД (RLS)**.

### 3.3. Сколько “досок по умолчанию” может быть
Для одного пользователя допускается:
- **0 или 1** доска по умолчанию.

То есть отсутствие выбранной доски по умолчанию — допустимое состояние.

### 3.4. Источник истины “доски по умолчанию”
“Доска по умолчанию” хранится **в базе данных**, а не в `localStorage`, чтобы:
- сохраняться между устройствами/браузерами;
- быть доступной серверу для редиректов.

Текущий механизм “последняя открытая доска” через `localStorage` **не влияет** на поведение DOIT и редиректы после внедрения этой спецификации.

## 4. Изменения в модели данных

### 4.1. `profiles.default_board_id`
Таблица `profiles` расширяется полем:
- `default_board_id` (uuid, `NULL` допустим) — ссылка на доску по умолчанию пользователя.

Ограничения поля:
- внешний ключ на `boards.id`;
- при удалении доски поле автоматически очищается: `ON DELETE SET NULL`.

### 4.2. RLS-ограничение на установку default_board_id
Для обновления своего профиля пользователь может установить `default_board_id` только если:
- `default_board_id IS NULL`, или
- пользователь является участником этой доски (существует строка в `board_members` с `board_id = default_board_id` и `user_id = auth.uid()`).

Попытка установить `default_board_id` на доску, где пользователь не участник, должна быть запрещена на уровне БД.

### 4.3. Политики безопасности (RLS), необходимые для фичи
Обязательные изменения политик RLS:
- **Удаление досок**: должна существовать политика `DELETE` на таблицу `boards`, разрешающая удаление строки доски только если:
  - `boards.owner_user_id = auth.uid()`, или
  - `public.is_system_admin()` возвращает `true`.
- **Установка default_board_id**: политика `UPDATE` на таблицу `profiles` для “обновления своего профиля” должна включать проверку из раздела 4.2 (иначе пользователь сможет выставить доску по умолчанию на чужую/недоступную доску).

## 5. Изменения на экране “Мои доски” (`/boards`)

### 5.0. Требуемые операции (контракты без привязки к реализации)
Экран `/boards` должен иметь возможность выполнить следующие операции.

#### 5.0.1. Загрузка данных для списка “Мои доски”
Операция возвращает:
- `boards[]`: список доступных пользователю досок (как сейчас: только те, где он участник).
  - обязательные поля на элементе: `id`, `name`, `created_at`;
  - дополнительные поля для UI:
    - `can_rename` (boolean) — `true`, если пользователь может переименовывать эту доску;
    - `can_delete` (boolean) — `true`, если пользователь может удалить эту доску (owner/system admin).
- `default_board_id` (uuid | null): текущая доска по умолчанию пользователя.

Правила вычисления:
- `can_rename` вычисляется по праву `board.rename` для этой доски.
- `can_delete` вычисляется как `(boards.owner_user_id = auth.uid()) OR (public.is_system_admin() = true)`.

#### 5.0.2. Переименование доски
Вход:
- `board_id` (uuid)
- `name` (string)

Правила:
- `name` нормализуется через `trim`.
- длина 1..100.
- операция должна быть запрещена без права `board.rename` (см. раздел 3.2).

Выход:
- успех, либо ошибка с сообщением из раздела 8.1.

#### 5.0.3. Удаление доски
Вход:
- `board_id` (uuid)

Правила:
- операция разрешена только owner/system admin (раздел 3.2 и 4.3).

Выход:
- успех, либо ошибка с сообщением из раздела 8.2.

#### 5.0.4. Установка/снятие “доски по умолчанию”
Вход:
- `default_board_id` (uuid | null)

Правила:
- если значение не `NULL`, пользователь обязан быть участником доски (раздел 4.2).

Выход:
- успех, либо ошибка с сообщением из раздела 8.3.

### 5.1. Список досок (новые элементы в строке доски)
Каждая строка списка досок должна содержать:
- **Название доски** (ссылка на `/boards/{boardId}`).
- **Флажок “По умолчанию”**:
  - отображается как checkbox;
  - отмечен, если `profiles.default_board_id = board.id`.
- **Меню действий** (или набор кнопок) справа:
  - “Переименовать” (если доступно по правам);
  - “Удалить” (если доступно по правам).

### 5.2. Правила отображения доступности действий в UI
Для каждой доски UI должен знать 2 булевых признака:
- `can_rename` — пользователь может переименовывать доску (право `board.rename`);
- `can_delete` — пользователь может удалять доску (owner или system admin).

Если признак `false`, действие не отображается в UI.

Если UI по каким-либо причинам отображает действие, но операция отклонена БД (например, права изменились), пользователь видит понятную ошибку (раздел 8).

### 5.3. Переименование доски (UX)
Сценарий:
- пользователь нажимает “Переименовать” у конкретной доски;
- открывается модальное окно **“Переименовать доску”** с полем ввода;
- поле заполнено текущим названием;
- кнопки: “Сохранить”, “Отмена”.

Правила ввода:
- перед валидацией применяется `trim`;
- длина: **1..100** символов после `trim`;
- если после `trim` пусто — “Сохранить” недоступна, показывается ошибка.

Правило сохранения:
- при “Сохранить” выполняется обновление имени доски;
- после успеха:
  - модалка закрывается;
  - в списке отображается новое имя без перезагрузки страницы (или с гарантированным обновлением данных).

### 5.4. Удаление доски (UX)
Сценарий:
- пользователь нажимает “Удалить” у конкретной доски;
- открывается модальное окно подтверждения **“Удалить доску?”**.

Требования к подтверждению:
- модалка содержит предупреждение: “Доска будет удалена для всех участников. Данные восстановить нельзя.”
- чтобы подтвердить удаление, пользователь должен:
  - ввести точное название доски в поле подтверждения (case-sensitive не требуется; сравнение выполняется после `trim`);
  - кнопка “Удалить” активна только при совпадении.

После подтверждения:
- выполняется удаление доски;
- после успеха:
  - доска исчезает из списка;
  - если удалённая доска была доской по умолчанию пользователя, `profiles.default_board_id` становится `NULL` автоматически;
  - пользователь остаётся на `/boards`.

## 6. “Доска по умолчанию” (UX и правила)

### 6.1. Переключение флажка
Поведение checkbox “По умолчанию”:
- если пользователь ставит флажок на доске A:
  - `profiles.default_board_id` становится равным `A.id`;
  - для всех остальных досок в списке флажок становится снятым (в UI это отображается немедленно).
- если пользователь снимает флажок на текущей доске по умолчанию:
  - `profiles.default_board_id` становится `NULL`.

### 6.2. Состояния и тексты
Если `profiles.default_board_id IS NULL`:
- ни у одной доски флажок не отмечен;
- приложение ведёт себя как “нет доски по умолчанию” (раздел 7).

### 6.3. Ошибки при сохранении default_board_id
Если изменение отклонено БД (например, пользователь потерял доступ к доске):
- UI обязан откатить состояние чекбокса к исходному;
- показать ошибку “Не удалось установить доску по умолчанию. Нет доступа к этой доске.”

## 7. Навигация и редиректы

### 7.1. Единая точка входа для “перейти в приложение”
В приложении должен существовать маршрут **`/go`** (или эквивалентный единый маршрут), который:
- если пользователь **не авторизован** → перенаправляет на `/` (лендинг);
- если пользователь **авторизован**:
  - если `profiles.default_board_id` задан и доска существует → перенаправляет на `/boards/{default_board_id}`;
  - иначе → перенаправляет на `/boards`.

### 7.2. Клик по DOIT в верхнем меню
Клик по логотипу/кнопке **DOIT** всегда ведёт на маршрут `/go` (а не на `/boards` и не на “последнюю открытую доску”).

### 7.3. Заход на главную страницу приложения (`/`)
Если пользователь авторизован и открывает `/`, он должен быть перенаправлен:
- на `/boards/{default_board_id}`, если доска по умолчанию задана;
- иначе на `/boards`.

### 7.4. Страница `/boards` не должна автоперенаправлять
Открытие `/boards` всегда показывает список “Мои доски” (даже если доска по умолчанию существует), чтобы пользователь мог управлять досками и настройкой по умолчанию.

## 8. Ошибки и сообщения (фиксированные)

### 8.1. Переименование
- Пустое название: “Укажите название доски.”
- Слишком длинное: “Название не длиннее 100 символов.”
- Нет прав: “Нет права переименовывать эту доску.”
- Ошибка сервера/БД: “Не удалось переименовать доску. Повторите попытку.”

### 8.2. Удаление
- Нет прав: “Нет права удалять эту доску.”
- Ошибка сервера/БД: “Не удалось удалить доску. Повторите попытку.”

### 8.3. Доска по умолчанию
- Нет доступа к выбранной доске: “Не удалось установить доску по умолчанию. Нет доступа к этой доске.”
- Ошибка сервера/БД: “Не удалось сохранить доску по умолчанию. Повторите попытку.”

## 9. Нефункциональные требования
- Любая операция (переименование, удаление, установка по умолчанию) должна иметь состояние “в процессе”:
  - кнопки/чекбоксы блокируются на время запроса;
  - повторный клик не должен создавать дубликат операции.
- После успешных операций список досок должен отображать актуальные данные без ручного обновления страницы.

## 10. Критерии приёмки
Считается выполненным, если:
1. На `/boards` у каждой доски есть:
   - ссылка на доску,
   - чекбокс “По умолчанию”,
   - действия “Переименовать” и “Удалить” (показываются только при наличии прав).
2. Пользователь с правом `board.rename` может переименовать доску, и новое имя отображается в списке.
3. Пользователь без `board.rename` не видит “Переименовать”; при попытке обхода получает ошибку “Нет права переименовывать эту доску.”
4. Владелец доски может удалить доску через подтверждение вводом названия; доска исчезает у всех участников.
5. Пользователь без прав удаления не видит “Удалить”; при попытке обхода получает ошибку “Нет права удалять эту доску.”
6. Пользователь может:
   - назначить любую свою доску доской по умолчанию,
   - снять назначение, получив состояние “по умолчанию не выбрано”.
7. При открытии `/` авторизованный пользователь перенаправляется:
   - на доску по умолчанию (если задана),
   - иначе на `/boards`.
8. При клике на DOIT авторизованный пользователь попадает:
   - на доску по умолчанию (если задана),
   - иначе на `/boards`.

## 11. Что не входит в задачу
- “Покинуть доску” (удаление только своего членства без удаления доски).
- Выбор нескольких досок по умолчанию.
- Использование “последней открытой доски” для редиректов.

## 12. Статус вопросов
Открытых вопросов для этой спецификации нет.
