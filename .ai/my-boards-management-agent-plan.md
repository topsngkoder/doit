# План реализации "Мои доски" (для AI-агента)

Основано на `.ai/my-boards-management-specification.md`.

Цель плана: дать исполняемую декомпозицию для приведения экрана `/boards`, навигации приложения и БД к спецификации "переименование, удаление, доска по умолчанию" с детальным трекером задач, порядком выполнения и критериями готовности для каждого шага.

## 0) Входные решения (зафиксировано спецификацией)
- У пользователя может быть `0..1` доска по умолчанию.
- Источник истины для доски по умолчанию: `profiles.default_board_id` в БД, не `localStorage`.
- `profiles.default_board_id`:
  - `uuid`, `NULL` допустим;
  - FK на `boards.id`;
  - при удалении доски очищается через `ON DELETE SET NULL`.
- Переименование доски разрешено только при праве `board.rename`.
- Удаление доски разрешено только:
  - владельцу доски (`boards.owner_user_id = auth.uid()`), или
  - системному администратору (`public.is_system_admin()`).
- Установка доски по умолчанию разрешена только для доски, где пользователь является участником.
- Экран `/boards` всегда остаётся экраном управления списком досок и не автоперенаправляет на дефолтную доску.
- Должен существовать единый маршрут входа `/go`:
  - неавторизованный пользователь -> `/`;
  - авторизованный пользователь с `default_board_id` -> `/boards/{id}`;
  - авторизованный пользователь без `default_board_id` -> `/boards`.
- Клик по логотипу/кнопке `DOIT` ведёт на `/go`.
- Заход авторизованного пользователя на `/` тоже должен приводить к логике `/go`.
- Текущий механизм `last opened board` через `localStorage` не должен влиять на поведение редиректов "в приложение".
- На `/boards` каждая строка доски должна содержать:
  - ссылку на доску;
  - checkbox `По умолчанию`;
  - действие `Переименовать`, если `can_rename = true`;
  - действие `Удалить`, если `can_delete = true`.
- Ошибки и тексты фиксированы спецификацией и не должны быть перепридуманы во время реализации.

## 1) Текущий технический контекст проекта

### 1.1. Что уже есть в коде
- `web/src/app/boards/page.tsx`
  - маршрут `/boards`;
  - проверяет сессию;
  - принимает инвайты;
  - грузит список досок через `from("boards").select("id, name, created_at")`;
  - рендерит только список ссылок и форму создания доски.
- `web/src/app/boards/actions.ts`
  - содержит только создание доски через RPC `create_board_with_defaults`.
- `web/middleware.ts`
  - неавторизованных на `/boards` отправляет на `/login`;
  - авторизованного пользователя на `/` или `/login` ведёт на `/boards`;
  - логики `default_board_id` не содержит.
- `web/src/components/doit-logo-link.tsx`
  - использует `localStorage` ключ `doit:last-opened-board-id`;
  - ведёт либо на `/boards/{id}`, либо на `/boards`.
- `web/src/components/last-opened-board-tracker.tsx`
  - записывает последнюю открытую доску в `localStorage`.
- `web/src/app/layout.tsx`
  - выводит `DoitLogoLink`;
  - ссылка "Мои доски" ведёт на `/boards`.
- `web/src/app/login/LoginForm.tsx`
  - после логина ведёт на `/boards`.
- `web/src/app/signup/actions.ts`
  - после регистрации ведёт на `/boards`.

### 1.2. Что уже есть в БД и авторизации
- В `boards` уже есть `owner_user_id`.
- В `board_members` уже есть membership-модель и признак owner membership.
- В проекте уже есть helper-функции:
  - `public.is_system_admin()`;
  - `public.has_board_permission(...)`.
- В RLS уже есть `SELECT` для `boards` через `board.view`.
- В RLS уже есть `UPDATE` для `boards`, завязанный на `board.rename` / `board.change_background`.
- Явной `DELETE` policy для `boards`, соответствующей новой спецификации, сейчас нет.
- В `profiles` сейчас нет `default_board_id`.
- Текущая policy обновления `profiles` не проверяет допустимость `default_board_id` через membership в `board_members`.

### 1.3. Явные расхождения с целевым состоянием
- Нет поля `profiles.default_board_id`.
- Нет RLS-ограничения на установку дефолтной доски только для участника этой доски.
- Нет `DELETE` policy для удаления доски owner/admin.
- Нет маршрута `/go`.
- Редирект с `/` не учитывает дефолтную доску.
- `DOIT` использует `localStorage`, хотя должен использовать `/go`.
- `/boards` не загружает `default_board_id`, `can_rename`, `can_delete`.
- На `/boards` нет checkbox "По умолчанию".
- На `/boards` нет rename/delete UI и соответствующих операций.
- Нет UX с модалкой переименования и модалкой удаления с подтверждением названием.
- Нет проводки фиксированных ошибок спецификации в пользовательский сценарий.

## 2) Стратегия выполнения
- Сначала зафиксировать модель данных и RLS, потому что UI зависит от допустимых операций в БД.
- Затем собрать серверные контракты для `/boards` и `/go`, чтобы фронтенд опирался на единый источник истины.
- После этого реализовать `/boards` UI:
  - checkbox `По умолчанию`;
  - действия `Переименовать` и `Удалить`;
  - состояния загрузки и откат optimistic/local state при ошибке.
- Затем перевести навигацию приложения на `/go` и убрать влияние `localStorage` на entry redirect.
- В конце провести приёмку по сценариям прав, UX и редиректов.

Порядок внедрения:
1. Схема БД.
2. RLS и SQL-ограничения.
3. Серверная загрузка данных `/boards`.
4. Операции rename / delete / set-default.
5. UI списка и модалки.
6. `/go`, `/`, логотип, login/signup redirects.
7. Удаление влияния legacy `last opened board`.
8. Проверка и приёмка.
